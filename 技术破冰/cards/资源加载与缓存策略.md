# 资源加载与缓存策略

## 1. 它属于哪个知识板块？

```txt
前端性能优化
→ 加载性能
→ 资源发现 / 请求优先级 / 压缩 / 缓存 / CDN
```

==资源优化的目标是让关键资源**更早发现、更快下载、更少重复下载**。==

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

==**资源加载与缓存策略**解决的是“**页面请求太多、资源太大、关键资源发现太晚、重复下载太多**”的问题。==

浏览器不是无限并发下载所有资源。HTML、CSS、JS、图片、字体、接口、第三方脚本都有发现时间、优先级、连接成本和缓存策略。关键资源慢一步，首屏就慢一截。

==优化资源加载的核心是把资源分成**关键资源、可延后资源、可缓存资源、第三方资源**，分别处理发现时机、优先级、体积和缓存。==

### 2.2 核心流程

```txt
打开 Network 面板
→ 看请求数量、大小、优先级、瀑布图
→ 找首屏关键资源
→ 找发现太晚的资源
→ 找体积异常资源
→ 找没有缓存或缓存策略错误的资源
→ 优化 preload / preconnect / compression / cache-control / CDN
→ 验证重复访问和弱网表现
```

### 2.3 关键词清单

1. critical resource：首屏渲染必须资源。
2. preload scanner：浏览器在解析 HTML 时提前扫描资源的机制。
3. preload：提前加载当前页面一定需要的关键资源。
4. preconnect：提前建立 DNS、TCP、TLS 连接。
5. dns-prefetch：提前做 DNS 解析。
6. prefetch：为空闲时加载未来可能需要的资源。
7. fetchpriority：调整资源请求优先级。
8. gzip / brotli：传输压缩。
9. Cache-Control：HTTP 缓存策略。
10. immutable asset：带 hash 的静态资源，适合长缓存。
11. CDN：内容分发网络，让静态资源离用户更近。
12. third-party script：第三方脚本，常见性能风险来源。

### 2.4 一句面试版

==资源加载优化的核心是按重要性管理资源：首屏关键资源要**尽早发现和高优先级下载**，非关键资源要**延后和懒加载**，带 hash 的静态资源要**长缓存**，跨域关键资源可用 **preconnect / preload** 辅助，最终用 Network 瀑布图和真实用户数据验证。==

### 2.5 最小 demo / 最小案例

关键字体或图片可以预加载：

```html
<link rel="preload" href="/fonts/app.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/hero.avif" as="image" fetchpriority="high">
```

跨域关键资源可预连接：

```html
<link rel="preconnect" href="https://cdn.example.com" crossorigin>
```

静态资源缓存策略：

```txt
/assets/app.8f3a2.js
Cache-Control: public, max-age=31536000, immutable

/index.html
Cache-Control: no-cache
```

### 2.6 常见优化手段

1. JS：代码分割、删除无用依赖、延后非关键脚本。
2. CSS：拆出关键 CSS，减少阻塞首屏的大 CSS。
3. 图片：合适尺寸、现代格式、懒加载非首屏图。
4. 字体：减少字重和字符集，必要时 preload。
5. 第三方脚本：异步加载、延后加载、定期审计。
6. 缓存：HTML 短缓存或协商缓存，hash 静态资源长缓存。
7. CDN：静态资源走 CDN，注意刷新和回滚策略。

## 3. 选择题自测

### Q1

资源加载优化的关键是什么？

A. 让关键资源更早发现、更快下载，非关键资源延后
B. 删除所有 CSS
C. 所有资源都 preload
D. 所有接口都不缓存

答案：A

### Q2

带 hash 的 JS/CSS 静态资源适合什么缓存？

A. 长缓存
B. 永远 no-store
C. 每秒重新下载
D. 只能内存缓存

答案：A

### Q3

`preconnect` 主要优化什么？

A. 提前建立跨域连接成本
B. 自动压缩图片
C. 自动删除 JS
D. 自动修复接口错误

答案：A

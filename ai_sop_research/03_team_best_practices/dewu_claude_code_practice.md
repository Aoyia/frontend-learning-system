- **主题**：团队落地实践、多 Agent 协作工作流

---

## 核心现成问答与方案总结

### 1. 团队落地 AI 辅助编程时，如何通过 SOP 规范对话逻辑？
自然语言存在多义性，直接与 AI 对话容易产生冗余或错误代码。得物团队提出**三阶段对话模型（SOP）**：
- **阶段一：需求定义** —— 使用“用户故事 + 验收标准（Acceptance Criteria）”结构化表达需求，保证需求边界清晰。
- **阶段二：边界明确** —— 严格区分“必须遵守的技术约束”（如分层架构规范、日志要求、鉴权中间件）和“建议参考的成熟代码”（如已有的状态机实现等）。
- **阶段三：迭代反馈** —— 采用“小步快跑”的原则，单次对话只专注于单个子模块。AI 在生成代码前必须主动暂停（例如数据库表结构设计完成后，先请开发者核对确认，无误后再开始生成配套的 Service 与 Controller 代码）。

### 2. 团队如何设计 AI 编程助手的 System Prompt（护栏配置）？
为了确保开发规范落地，通常会在项目根目录放置类似 `CLAUDE.md` 的规则文件。但得物团队指出，编写该规则文件时要避免“信息过载”。
- **推荐方案**：把规则文件控制在 200 字以内。它应该充当“护栏”而非“详尽的开发手册”。主要用来规避团队中最常出现的低级错误或禁忌（如禁止使用某类 API、要求做非空校验等）。更具体的业务领域知识应当存放在独立的文档中，并在需要时通过指引让 AI 动态读取。

### 3. 多 Agent（子代理）在团队协作中如何保持状态一致？
在复杂的研发场景下，单 Agent 难以胜任全流程。如果多个 Agent 相互对话，可能会因为信息冗余而造成通信混乱。
- **推荐方案**：采用**“中间产物驱动协作”模式**。所有的 AI Agent（包括方案架构师 Agent、代码实现 Agent、代码审查 Agent等）都不直接进行相互通信，而是围绕同一个“技术方案文档/中间产物”进行增量更新和消费。例如：架构师 Agent 编写方案 -> 代码 Agent 消费该方案并实现代码 -> 审查 Agent 依据方案和代码进行审查。以此确保各 Agent 状态一致。

# 请求-响应全链路

## 1. 它属于哪个知识板块？

```txt
后端基础
→ 接口与协议
→ 请求-响应模型
→ 一次 HTTP 请求从发出到回来
```

==它**不是讲某个框架**，而是回答一个最基础的问题：你在浏览器里 `fetch` 一个 URL，请求**离开浏览器之后到底经历了什么**，最后又是怎么把响应送回来的。==

这一站是后端的"破冰起手卡"。这一张写完，你会突然听懂大部分后端术语。

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

==**请求-响应全链路**解决的是“**前端工程师只看到 fetch 的两端，看不到中间**”的问题。==

中级以下前端常见的认知是：**fetch 是一个魔法函数，传 URL 进去，过几百毫秒拿到 JSON**。但凡线上出问题，就只能猜：是网络？是后端？是 CORS？是 cookie？是缓存？

==所以第一站的核心**不是任何具体技术**，而是把**“一个请求”这件事**拆成可以指认的环节：**URL 解析、DNS、TCP、TLS、HTTP 报文、反向代理、应用服务、业务逻辑、数据库、响应回传**。==

### 2.2 核心流程

```txt
浏览器调用 fetch(url)
→ 解析 URL（协议、域名、端口、路径、查询参数）
→ DNS 查询：域名 → IP 地址
→ 建立 TCP 连接（三次握手，连到 IP:port）
→ 如果是 https，再做 TLS 握手（协商加密 + 验证证书）
→ 浏览器发送 HTTP 请求报文（method、headers、body）
→ 请求到达服务器机房
→ 反向代理（如 Nginx）按规则转发到后端服务
→ 后端服务（Node / Java / Go / ...）路由到对应 handler
→ handler 执行业务逻辑（读 / 写数据库、调其它服务、加工数据）
→ 组装 HTTP 响应报文（status、headers、body）
→ 沿原路返回浏览器
→ 浏览器按状态码与 headers 处理响应（解析 JSON、写 cookie、缓存等）
```

### 2.3 关键词清单

1. URL：统一资源定位符，结构是 `protocol://host:port/path?query#hash`。
2. DNS：域名解析系统，把人类看的域名换成机器用的 IP 地址。
3. IP / port：IP 是一台机器在网络中的地址，port 是这台机器上某个进程监听的端口号。一台机器上可以有很多端口同时被不同进程监听。
4. TCP：一种**面向连接、可靠**的传输协议，HTTP/1.x 与 HTTP/2 跑在它上面。
5. TLS / HTTPS：在 TCP 之上加密，HTTPS = HTTP over TLS，要做证书验证与密钥协商。
6. HTTP method：GET / POST / PUT / PATCH / DELETE 等，表达对资源的不同操作语义。
7. status code：1xx 信息、2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务端错误。
8. headers：键值对元信息，例如 `Content-Type`、`Authorization`、`Cookie`、`Cache-Control`、`Accept`。
9. body：请求体或响应体，可能是 JSON、表单、二进制等。
10. CORS：浏览器同源策略下的跨域机制，由响应头 `Access-Control-*` 决定是否允许。
11. reverse proxy：反向代理（Nginx、Caddy 等），对外接收请求，对内转发到一组后端服务。
12. handler / route：后端服务里负责处理某个 path + method 的函数。
13. middleware：在 handler 前后执行的通用逻辑，例如日志、鉴权、限流、错误处理。
14. database：持久化存储，常见是关系型（MySQL、Postgres）或键值（Redis）。
15. response：响应报文，由 status + headers + body 组成，结构和请求对称。

### 2.4 一句面试版

==一次 HTTP 请求从浏览器发出后，会经历 **URL 解析 → DNS 查 IP → TCP 建连 → TLS 握手 → 发送 HTTP 报文 → 反向代理转发 → 后端 handler 处理（含业务、数据库、外部调用）→ 返回 HTTP 响应**；任何线上问题，都可以被定位到这条链路里的某一环。==

### 2.5 最小 demo / 最小案例

#### 用 curl 看清一个真实请求

```bash
curl -v https://httpbin.org/get
```

输出大致包含：

```txt
* Trying 3.219.x.x:443...                # 已经把域名解析成 IP，开始建 TCP 连接
* Connected to httpbin.org (3.219.x.x) port 443
* TLSv1.3 (OUT), TLS handshake, ...      # TLS 握手
> GET /get HTTP/2                        # 发出去的请求行
> Host: httpbin.org                      # 请求头
> User-Agent: curl/8.1.2
> Accept: */*
>
< HTTP/2 200                             # 服务器返回状态行
< content-type: application/json         # 响应头
< content-length: 287
<
{ "args": {}, ... }                      # 响应体
```

`-v` 让 curl 把上面的整条链路打印出来。这是把"抽象的请求过程"变成"看得见的文字"的最快方式。

#### 用浏览器 DevTools 看同一件事

```txt
打开 Network 面板
→ 选中一个请求
→ Headers：看 Request Line、Request Headers、Response Headers、Status
→ Timing：看 DNS、Initial connection、SSL、Waiting (TTFB)、Content Download 各阶段耗时
→ Response：看响应体
```

==**Timing 面板的每一段时间，对应的就是上面 2.2 流程图里的一段。**==

### 2.6 前端类比：你已经会的那部分

| 你已经熟悉的 | 它在全链路里的位置 |
| --- | --- |
| `fetch(url, options)` | 链路最前端，发起请求 |
| URL 里的 `https://api.x.com:8443/users?id=1` | URL 解析后会拿到 protocol、host、port、path、query |
| `Content-Type: application/json` | 请求头之一，告诉服务端 body 怎么解析 |
| `Set-Cookie` 响应头 | 服务端写回客户端的状态，浏览器自动管理 |
| CORS 报错 | 浏览器在响应阶段根据 `Access-Control-*` 决定要不要拦下 |
| 接口 504 | 反向代理超时；通常是后端 handler / 数据库 / 下游服务卡住 |
| 接口 502 | 反向代理拿不到后端响应；通常是后端服务挂了或没起来 |
| 接口 401 / 403 | 鉴权 / 权限失败，业务层在 handler 里返回的 |
| 接口 500 | 后端 handler 里抛了未捕获异常 |

### 2.7 常见问题该往链路哪一段看？

```txt
打开慢，第一字节就慢
→ DNS / TCP / TLS / 反向代理 / 后端 TTFB

页面能开但接口 4xx
→ 业务逻辑、参数、鉴权、CORS

接口 5xx
→ 后端服务异常、依赖（数据库 / 第三方服务）异常

偶尔失败、其它人没问题
→ 客户端网络、CDN 节点、运营商劫持

CORS 报错
→ 服务端响应头是否正确（Access-Control-Allow-Origin、Allow-Methods 等）
```

### 2.8 是否值得深入？

非常值得，是后端的**地基**。优先顺序：

1. 先打通这条链路的命名与位置感（看到 504 知道指向哪一段）。
2. 再掌握 HTTP 报文结构、状态码语义、常用头。
3. 然后理解 DNS、TCP、TLS 这一层在做什么（不需要会推协议，知道职责即可）。
4. 接着理解反向代理与后端服务的关系。
5. 最后再深入 HTTP/2、HTTP/3、WebSocket、SSE、长连接这些进阶协议。

优先看官方资料：MDN HTTP、MDN Fetch、curl 文档、Chrome DevTools Network。

## 3. 选择题自测

### Q1

下面哪种说法最准确？

A. fetch 是浏览器的一个魔法函数，无法被拆解
B. 一次 HTTP 请求会依次经历 URL 解析、DNS、TCP、TLS（如 https）、HTTP 报文、反向代理、后端 handler、数据库、响应回传
C. 所有 HTTP 请求都不需要 DNS
D. HTTP 请求只有 GET 和 POST

答案：B

解析：B 是这一站的核心结论。后端排查问题的第一步就是把现象映射到链路某一段。

### Q2

DNS 的作用是什么？

A. 把域名换成 IP 地址
B. 给请求加密
C. 替代 TCP
D. 自动写后端代码

答案：A

解析：DNS 是"域名 → IP" 的查询系统，是 fetch 之前必经的一步。

### Q3

接口返回 502，最值得先怀疑哪一段？

A. 浏览器解析 JSON 的代码
B. 反向代理与后端服务之间——后端服务可能挂了或没起来
C. 用户的鼠标
D. 浏览器主题色

答案：B

解析：502 Bad Gateway 通常是反向代理拿不到后端响应。

### Q4

下列哪一项不属于响应报文的组成？

A. status code
B. headers
C. body
D. DNS 查询结果

答案：D

解析：DNS 在请求发出前发生，不属于 HTTP 响应报文本身。

### Q5

浏览器 DevTools 的 Timing 面板里，"Waiting (TTFB)"这一段最能反映什么？

A. 服务端处理请求并返回首字节的耗时
B. 用户网速
C. CSS 解析速度
D. JS 执行速度

答案：A

解析：TTFB 反映从发出请求到收到第一个字节的时间，主要受后端处理速度与网络往返影响。

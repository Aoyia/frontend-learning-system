# 开发服务器

## 1. 陌生技术出现后的学习顺序

### 1.1 它解决什么问题？

==**开发服务器**解决的是“开发时如何**快速看到代码变化**”的问题。==

生产构建关注产物体积、缓存、兼容性和部署稳定性；开发环境更关注启动快、修改快、报错快、代理接口方便。如果每改一行代码都要完整构建一次，开发反馈会非常慢。

==所以 dev server 的核心价值是用**本地 HTTP 服务**承载源码和转换结果，**按需处理浏览器请求**，并通过 **HMR** 尽量只更新变化的模块。==

### 1.2 核心流程

```txt
启动 dev server
→ 读取项目配置
→ 浏览器请求 index.html
→ 按需解析和转换模块
→ 处理裸模块导入和依赖预构建
→ 建立 WebSocket 连接
→ 文件变化后触发 HMR
→ 浏览器局部更新或整页刷新
```

### 1.3 关键词清单

1. dev server：本地开发 HTTP 服务，例如 Vite dev server、webpack-dev-server。
2. native ESM：浏览器原生 ES module 加载能力，Vite 开发环境会围绕它做按需加载。
3. bare import：裸模块导入，例如 `import { ref } from 'vue'`，浏览器不能直接识别，需要 dev server 改写。
4. dependency pre-bundling：依赖预构建，把第三方依赖提前转换成更适合浏览器加载的 ESM。
5. transform：按需转换源码，例如 TypeScript、JSX、Vue SFC、CSS。
6. HMR：Hot Module Replacement，热模块替换，尽量在不刷新整页、不丢状态的情况下更新模块。
7. WebSocket：dev server 和浏览器之间的通信通道，用来推送更新消息。
8. proxy：开发代理，把前端本地请求转发到后端接口，解决本地跨域和环境切换问题。
9. full reload：当变化无法局部热更新时，退回整页刷新。
10. overlay：浏览器错误浮层，把编译错误或运行错误直接展示到页面上。

### 1.4 一句面试版

==开发服务器通过**本地 HTTP 服务**按需处理源码和依赖请求，并用 **WebSocket** 通知浏览器做 **HMR**，让开发环境不必每次完整构建，也能快速看到代码变化。==

### 1.5 最小 demo / 最小案例

Vite 项目里最小开发命令通常是：

```json
{
  "scripts": {
    "dev": "vite --host 0.0.0.0",
    "preview": "vite preview"
  }
}
```

接口代理可以这样配置：

```ts
import { defineConfig } from 'vite'

export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3000',
        changeOrigin: true
      }
    }
  }
})
```

这表示浏览器请求 `/api/users` 时，开发服务器会把请求转发到 `http://localhost:3000/api/users`。

### 1.6 HMR 怎么理解？

普通刷新是：

```txt
文件变化
→ 重新加载整个页面
→ 应用状态丢失
```

HMR 是：

```txt
文件变化
→ dev server 找到受影响模块
→ 通过 WebSocket 通知浏览器
→ 浏览器拉取新模块
→ 框架或模块自己接收更新
→ 尽量保留当前页面状态
```

Vite 的 HMR API 暴露在 `import.meta.hot` 上。业务开发通常不用手写 HMR，Vue SFC、React Fast Refresh 这类框架插件会帮你处理。

### 1.7 是否值得深入？

值得深入，尤其是排查本地开发问题时。优先顺序：

1. 先理解 dev server 和 production build 不是一回事。
2. 再理解浏览器请求源码时，dev server 如何按需转换。
3. 然后理解裸模块导入为什么需要改写和预构建。
4. 接着掌握 HMR、full reload、proxy、环境变量。
5. 最后再深入插件如何参与 transform、HMR 边界和依赖缓存失效。

优先看官方资料：Vite features、Vite HMR API、webpack-dev-server。

## 2. 选择题自测

### Q1

开发服务器最核心的价值是什么？

A. 替代生产部署
B. 提升本地开发反馈速度，按需处理模块并支持热更新
C. 自动编写测试用例
D. 删除构建工具

答案：B

解析：dev server 面向开发阶段，重点是启动快、修改反馈快。

### Q2

为什么浏览器不能直接处理 `import { ref } from 'vue'` 这种裸模块导入？

A. 浏览器不知道 `vue` 应该对应哪个具体 URL
B. 浏览器不支持任何 JavaScript
C. `vue` 只能在服务端运行
D. 因为代码没有压缩

答案：A

解析：浏览器原生 ESM 需要可访问的 URL，dev server 会把裸模块导入改写成实际可请求的地址。

### Q3

HMR 和整页刷新的关键区别是什么？

A. HMR 一定会删除所有状态
B. HMR 尽量只替换变化模块，并保留当前页面状态
C. HMR 只能用于后端服务
D. HMR 不需要浏览器

答案：B

解析：HMR 的目标是更小范围更新，失败或无法处理时才退回 full reload。

### Q4

开发代理 proxy 常用来解决什么问题？

A. 本地请求后端接口时的跨域和环境转发
B. 压缩图片体积
C. 生成 TypeScript 类型
D. 自动发布生产环境

答案：A

解析：dev server 的 proxy 可以让前端以本地域名请求，再由服务器转发到真实后端地址。

### Q5

下面哪个说法正确？

A. dev server 的结果一定等于生产构建结果
B. dev server 主要服务开发体验，生产上线仍应使用 build 产物
C. 有了 HMR 就不需要测试
D. 开发环境不需要处理任何依赖

答案：B

解析：开发服务器和生产构建目标不同，不能把本地 dev server 当成生产部署方案。

# 包管理器

## 1. 陌生技术出现后的学习顺序

### 1.1 它解决什么问题？

==**包管理器**解决的是“项目依赖**从哪里来、装什么版本、怎么让团队和 CI 装得一致**”的问题。==

前端项目不是只靠自己写的代码运行。它通常依赖 Vue、React、Vite、TypeScript、ESLint、测试框架和大量间接依赖。如果每个人安装出来的依赖树不一样，就容易出现“我这里能跑，你那里报错”。

==所以包管理器的核心**不是下载包**，而是围绕**依赖声明、版本解析、lockfile、安装策略、脚本命令和工作区管理**，建立一套**可复现的依赖体系**。==

### 1.2 核心流程

```txt
读取 package.json
→ 解析 dependencies / devDependencies
→ 根据 semver 范围选择版本
→ 写入或读取 lockfile
→ 下载依赖包
→ 生成 node_modules 或 PnP 映射
→ 暴露 bin 命令
→ 执行 npm scripts
```

### 1.3 关键词清单

1. `package.json`：项目依赖、脚本、包名、版本、入口、工作区等元信息。
2. dependencies：运行时依赖，代码运行时需要，例如 Vue、React。
3. devDependencies：开发依赖，开发、检查、构建时需要，例如 Vite、ESLint、TypeScript。
4. semver：语义化版本，例如 `^1.2.3`、`~1.2.3`、`1.2.3`，决定允许升级到什么范围。
5. lockfile：锁定完整依赖树，例如 `package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`。
6. `node_modules`：依赖安装目录，不同包管理器会用不同布局和 hoist 策略。
7. hoisting：依赖提升，把某些依赖放到更上层目录，方便复用，但也可能带来幽灵依赖。
8. ghost dependency：代码使用了没有在自己 `package.json` 声明的依赖，靠依赖提升“碰巧能跑”。
9. workspace：多包仓库能力，用来管理 Monorepo 里的多个 package。
10. scripts：统一命令入口，例如 `dev`、`lint`、`test`、`build`。

### 1.4 一句面试版

==包管理器通过 `package.json` **声明依赖**，通过 **lockfile 固定完整依赖树**，并用安装策略和 scripts 统一本地、团队、CI 的依赖安装与命令执行，核心目标是让项目依赖**可复现、可维护**。==

### 1.5 最小 demo / 最小案例

一个最小前端项目通常会有这些内容：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "dependencies": {
    "vue": "^3.5.0"
  },
  "devDependencies": {
    "vite": "^8.0.0",
    "typescript": "^5.9.0"
  }
}
```

本地新增依赖时：

```bash
npm install vue
```

CI 安装依赖时更适合使用：

```bash
npm ci
```

因为 `npm ci` 会要求 lockfile 存在，并且当 lockfile 与 `package.json` 不一致时直接失败，不会悄悄改 lockfile。

### 1.6 npm、pnpm、Yarn 怎么区分？

1. npm：Node 默认包管理器，生态通用，CI 中常用 `npm ci` 做干净安装。
2. pnpm：依赖存到 content-addressable store，再通过硬链接和符号链接组织项目依赖，节省磁盘空间，也更容易暴露未声明依赖。
3. Yarn：现代版本提供 Plug'n'Play，可不依赖传统 `node_modules` 查找方式，并能更严格地发现幽灵依赖。
4. 选型原则：个人项目 npm 足够；团队或 Monorepo 常见 pnpm；已有 Yarn PnP 体系的项目继续跟随项目约定。

### 1.7 是否值得深入？

值得深入，但先抓主线：

1. 先理解 `package.json` 和 lockfile 的分工。
2. 再理解 `npm install` 和 `npm ci` 的区别。
3. 然后理解 dependencies 和 devDependencies 的边界。
4. 接着看 node_modules 布局、hoisting、幽灵依赖。
5. 最后再深入 workspace、peerDependencies、依赖安全审计、私有 registry。

优先看官方资料：npm package.json、npm ci、pnpm motivation、Yarn Plug'n'Play。

## 2. 选择题自测

### Q1

lockfile 最核心的作用是什么？

A. 记录页面路由
B. 固定完整依赖解析结果，让不同环境尽量安装出一致依赖树
C. 替代源代码
D. 自动压缩图片

答案：B

解析：`package.json` 通常声明版本范围，lockfile 记录具体解析结果。

### Q2

CI 中为什么常用 `npm ci` 而不是 `npm install`？

A. `npm ci` 会自动写业务代码
B. `npm ci` 更适合干净、可复现的自动化安装，lockfile 不一致时会失败
C. `npm ci` 不需要 package.json
D. `npm ci` 会跳过所有依赖安装

答案：B

解析：`npm ci` 面向 CI、测试、部署这类自动化环境，不会改写 lockfile。

### Q3

ghost dependency 指的是什么？

A. 由于 package.json 锁定的包在远端 Registry 物理下线导致的安装失败依赖
B. 业务中导入了未在 package.json 依赖表显式声明的包，但在构建时因包管理器扁平化提升机制而碰巧能被解析引用
C. 那些在多包 Monorepo 构建链路中仅用作本地软链接导出的虚拟包
D. 浏览器在加载协商缓存静态资源时隐式生成的私有 DOM 模型引用包

答案：B

解析：幽灵依赖会让项目在不同安装策略或发布环境下突然失败。

### Q4

下面哪个依赖更适合放进 devDependencies？

A. 运行时代码直接 import 的 Vue
B. 用户浏览页面必须加载的组件库
C. 只在构建阶段使用的 TypeScript
D. 生产环境必须运行的 HTTP 客户端

答案：C

解析：TypeScript 通常用于开发和构建检查，不是浏览器运行时代码直接依赖。

### Q5

pnpm 节省磁盘空间的关键思路是什么？

A. 完全不下载依赖
B. 把依赖文件存入统一 store，再用链接复用
C. 把所有依赖写进一个 JS 文件
D. 删除 package.json

答案：B

解析：pnpm 的 store 和链接策略让多个项目可以复用相同依赖文件。

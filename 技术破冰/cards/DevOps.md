# DevOps

## 1. 它属于哪个知识板块？

DevOps 属于：

```txt
软件交付与运维
→ 开发协作 / 自动化交付 / 运行稳定性
→ CI/CD / Docker / K8s / 监控告警 / 发布回滚 / IaC
```

==**DevOps** 解决的是“开发只管写代码，运维只管上线和救火，导致交付慢、故障多、反馈断层”的问题。==

它不是某一个工具，也不是单纯的 CI/CD，更不是只属于后端或运维的岗位名称。更准确地说：

==DevOps 是一套把**开发、测试、部署、运行、监控、反馈**连成闭环的工程实践和协作方式，目标是让软件交付更快、更稳定、更可追踪。==

### 1.1 先和传统工作流对齐

如果按 [[传统软件开发工作流|传统软件开发工作流]] 看，一个软件通常会经历：

```txt
需求
→ 设计
→ 开发
→ 测试
→ 发布
→ 运维维护
```

传统流程的问题不是这些阶段本身错了，而是阶段之间经常形成“墙”：

```txt
开发写完扔给测试
→ 测试测完扔给发布
→ 发布上线扔给运维
→ 运维发现问题再找开发
```

==DevOps 不是删除需求、设计、开发、测试、发布和运维，而是把这些阶段之间的交接变成**自动化流水线、标准化环境、可观测反馈和共同责任**。==

所以可以这样理解：

```txt
传统工作流：按阶段把软件交付拆开
DevOps：把这些阶段重新连成一个快速反馈闭环
```

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

没有 DevOps 思路时，团队里经常会出现这些断层：

1. 开发写完代码后，不清楚怎么上线。
2. 测试环境、预发环境、生产环境配置不一致。
3. 发布靠人工手动操作，步骤多、容易漏。
4. 出故障后不知道哪个版本、哪个配置、哪次发布导致问题。
5. 线上有错误，但开发很晚才知道。
6. 回滚没有标准流程，只能临时救火。
7. 运维和开发互相甩锅：一个说代码有问题，一个说环境有问题。

==DevOps 的核心不是“让开发替代运维”，而是把**代码变更到线上运行**这条链路工程化：自动化、标准化、可观测、可回滚、可持续改进。==

### 2.2 核心闭环

DevOps 经常被画成一个无限循环：

```txt
Plan 计划
→ Code 编码
→ Build 构建
→ Test 测试
→ Release 发布准备
→ Deploy 部署
→ Operate 运行
→ Monitor 监控
→ Feedback 反馈回开发
```

这条链路的重点是：

1. 代码不是写完就结束，而是要稳定运行在线上。
2. 发布不是一次手工动作，而是可重复的流水线。
3. 监控不是运维私事，而是给开发反馈真实用户和真实系统状态。
4. 故障不是只追责，而是反过来改进测试、发布、回滚和监控。

==DevOps 真正关心的是：一次代码变更，能不能从提交到上线再到线上反馈，形成一条可靠闭环。==

### 2.3 四层心智模型

第一次理解 DevOps，不要从 Jenkins、K8s、云平台这些工具开始。先按四层看：

```txt
协作层：开发、测试、运维、平台、安全如何共同负责交付结果
自动化层：CI/CD 如何自动检查、测试、构建、发布
运行层：Docker / K8s / 云平台如何稳定运行应用
反馈层：监控、日志、告警、事故复盘如何把线上信息反馈给开发
```

这四层对应的问题是：

1. **协作层**：谁对上线质量和线上结果负责？
2. **自动化层**：哪些人工步骤可以变成流水线？
3. **运行层**：应用在哪些环境里运行，环境是否一致？
4. **反馈层**：上线后出问题，能不能及时发现、定位、恢复和改进？

==DevOps 的破冰关键是：它不是一个工具清单，而是一套围绕“交付闭环”的工程能力组合。==

## 3. DevOps 不等于什么？

### 3.1 不等于 CI/CD

CI/CD 是 DevOps 里很重要的自动化实践，但 DevOps 不只是一条流水线。

```txt
CI/CD：检查、测试、构建、部署自动化
DevOps：从协作方式到交付流程，再到线上运行反馈的一整套体系
```

可以这么理解：

==CI/CD 是 DevOps 的“交付发动机”，但 DevOps 还包括环境治理、发布策略、监控告警、故障响应、安全、权限、成本和团队协作。==

### 3.2 不等于 Docker / K8s

Docker 解决的是应用如何被打包成镜像并运行。

K8s 解决的是大量容器如何被调度、扩缩容、滚动发布和自愈。

DevOps 关心的是更上层的问题：

```txt
代码怎么进流水线
→ 产物怎么构建和保存
→ 镜像怎么发布
→ 环境怎么管理
→ 部署怎么回滚
→ 线上怎么监控
→ 故障怎么复盘
```

Docker 和 K8s 是 DevOps 实践中常见的技术底座，不是 DevOps 本身。

### 3.3 不等于 Azure DevOps

Azure DevOps 是微软的一套 DevOps 平台产品。

类似的平台还有：

1. GitHub Actions。
2. GitLab CI/CD。
3. [[Jenkins|Jenkins]]。
4. CircleCI。
5. Argo CD。

这些都是工具。DevOps 是工具背后的交付和运维思想。

## 4. 关键词清单

1. CI：持续集成，每次代码变更后自动检查、测试、构建。
2. CD：持续交付或持续部署，让构建产物处于可发布状态，甚至自动部署。
3. pipeline：流水线，把检查、测试、构建、发布等步骤自动化。
4. artifact：流水线产物，例如 `dist`、测试报告、sourcemap、Docker 镜像。
5. environment：环境，例如 dev、test、staging、production。
6. infrastructure as code：基础设施即代码，用代码描述服务器、网络、数据库、权限等资源。
7. container：容器，把应用和运行依赖封装成可运行单元。
8. orchestration：编排，管理大量容器的部署、调度、扩缩容和自愈。
9. observability：可观测性，通过 logs、metrics、traces 理解系统状态。
10. alerting：告警，当错误率、延迟、资源、可用性异常时通知负责人。
11. rollback：回滚，把线上系统恢复到上一个稳定版本或配置。
12. feature flag：功能开关，不重新发布也能控制功能是否对用户开放。
13. incident：线上事故，需要响应、止血、复盘和改进。
14. DORA metrics：衡量交付效能的指标，常见是部署频率、变更前置时间、变更失败率、恢复时间。
15. DevSecOps：把安全检查和安全治理前移到开发与交付链路中。

## 5. 它和你已经有的卡片怎么连？

```txt
DevOps
├─ [[传统软件开发工作流|传统软件开发工作流]]：理解需求、设计、开发、测试、发布、维护这些基础阶段
├─ [[CI-CD|CI/CD]]：把检查、测试、构建、部署自动化
├─ [[Jenkins|Jenkins]]：承载 CI/CD 流水线的自动化服务器
├─ [[发布策略|发布策略]]：灰度、金丝雀、蓝绿、回滚
├─ [[Docker 与容器化|Docker 与容器化]]：把应用打包成镜像并运行
├─ [[K8s 与容器编排|K8s]]：大规模容器编排、扩缩容、滚动发布和自愈
├─ 前端可观测性：错误监控、性能监控、日志、Trace ID
└─ 前端安全：把安全检查放进开发和流水线
```

对前端来说，DevOps 不是离你很远。你会在这些场景里碰到它：

1. PR 合并前自动跑 lint、typecheck、test、build。
2. 前端构建产物上传 CDN。
3. sourcemap 上传到监控平台。
4. 发布灰度，只给部分用户或租户开放。
5. 线上白屏率、接口错误率、性能指标异常后触发告警。
6. 发布失败后自动回滚旧版本。
7. 用 Docker 镜像交付 Node BFF 或前端静态资源服务。

==前端视角理解 DevOps，不是去背运维命令，而是理解“代码如何可靠地变成线上可运行系统，并在出问题时能被发现、定位和恢复”。==

## 6. 最小 demo / 最小案例

一个最小前端 DevOps 链路可以这样看：

```txt
开发者提交 PR
→ CI 安装依赖
→ 跑 lint / typecheck / test
→ 执行 npm run build
→ 保存 dist 或构建 Docker 镜像
→ 部署到测试环境
→ 人工审批或自动发布到生产
→ 发布后监控错误率、性能、接口失败率
→ 出问题回滚
```

对应到文件和平台：

```txt
package.json：定义本地和 CI 共用命令
lockfile：固定依赖版本
.github/workflows/*.yml：定义流水线
Dockerfile：定义运行镜像
nginx.conf：定义静态资源服务和路由回退
监控平台：收集错误、性能、日志和告警
发布平台：控制环境、审批、灰度和回滚
```

最小 `package.json`：

```json
{
  "scripts": {
    "lint": "eslint .",
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "build": "vite build"
  }
}
```

最小流水线关注点：

```txt
依赖安装是否可复现
→ 检查是否自动执行
→ 构建产物是否保存
→ 部署目标环境是否明确
→ 失败时是否停止发布
→ 上线后是否能观测和回滚
```

这就是 DevOps 的入门骨架。

## 7. 常见误解

### 7.1 “DevOps 就是运维”

不准确。传统运维更关注服务器、网络、进程、资源和稳定性。DevOps 关注的是开发到运行的整个交付闭环。

### 7.2 “DevOps 就是会 Jenkins”

不准确。Jenkins 是流水线工具。只会写 Jenkinsfile，但没有测试、产物、环境、权限、回滚和监控意识，不算真正理解 DevOps。

### 7.3 “有了 Docker 就是 DevOps”

不准确。Docker 只解决交付载体问题。如果发布仍然靠手工、监控缺失、事故不能回滚，DevOps 仍然没有做好。

### 7.4 “DevOps 是某一个人的职责”

不完全准确。可以有 DevOps 工程师或平台工程团队，但 DevOps 更重要的是团队责任边界：

```txt
开发对线上运行结果有责任
运维 / 平台对交付效率和稳定性有责任
测试 / 安全 / 产品都要进入反馈闭环
```

## 8. 和 SRE / 平台工程的关系

### 8.1 SRE

SRE 是 Site Reliability Engineering，站点可靠性工程。

可以先这样区分：

```txt
DevOps：强调开发和运维协作，打通交付与反馈闭环
SRE：更强调用工程方法管理可靠性，例如 SLO、错误预算、自动化运维、事故复盘
```

SRE 更偏“如何把系统稳定性用工程方法量化和治理”。

### 8.2 平台工程

平台工程更偏“给开发团队提供内部平台”，例如：

1. 一键创建服务。
2. 标准流水线模板。
3. 统一日志和监控。
4. 统一发布和回滚入口。
5. 权限、密钥、环境变量管理。

平台工程可以看作 DevOps 规模化后的组织形态之一：把重复的交付和运维能力沉淀成平台，减少每个业务团队重复造轮子。

## 9. 一句面试版

==DevOps 是一种把**开发、测试、部署、运行、监控和反馈**打通的工程实践，核心目标是通过**自动化流水线、标准化环境、可观测性和快速回滚**，让软件交付更快、更稳定、更可追踪；CI/CD、Docker、K8s、监控告警和发布策略都是 DevOps 里的具体实践。==

## 10. 是否值得深入？

值得。尤其当你开始接触：

1. CI/CD 流水线。
2. Docker 镜像。
3. Node BFF 部署。
4. 前端灰度发布。
5. sourcemap 上传。
6. 线上错误监控。
7. K8s、云平台、内部发布平台。

推荐深入顺序：

1. 先理解 DevOps 是交付与运行闭环，不是某个工具。
2. 再学 [[CI-CD|CI/CD]]，理解流水线和 artifact。
3. 然后学 [[Docker 与容器化|Docker 与容器化]]，理解镜像和容器。
4. 接着学 [[K8s 与容器编排|K8s 与容器编排]]，理解集群里的容器如何被编排。
5. 再学发布策略、回滚、feature flag。
6. 接着学可观测性：日志、指标、链路追踪、告警。
7. 最后再深入 IaC、GitOps、SRE、平台工程和 DevSecOps。

优先看官方资料：AWS What is DevOps、Microsoft Learn What is DevOps、Google Cloud DevOps capabilities、DORA metrics、Atlassian DevOps guide。

## 11. 选择题自测

### Q1

DevOps 最核心解决的问题是什么？

A. 让开发永远不需要写测试  
B. 打通开发、交付、运行和反馈，减少交付慢、环境不一致、故障难恢复的问题  
C. 替代所有产品经理  
D. 只负责买云服务器

答案：B

解析：DevOps 关注的是从代码变更到线上运行再到反馈的完整闭环。

### Q2

CI/CD 和 DevOps 的关系更准确的是？

A. CI/CD 是 DevOps 里的自动化交付实践之一  
B. CI/CD 和 DevOps 完全无关  
C. DevOps 只等于 CI/CD  
D. CI/CD 只能用于后端

答案：A

解析：CI/CD 是 DevOps 很核心的一部分，但 DevOps 还包括环境、发布、监控、回滚、安全和协作。

### Q3

Docker 在 DevOps 里通常扮演什么角色？

A. 把应用和运行依赖打包成镜像，作为可交付、可运行的产物  
B. 自动设计页面  
C. 替代所有监控  
D. 自动写业务需求

答案：A

解析：Docker 常用于标准化应用交付和运行环境。

### Q4

下面哪个更像 DevOps 里的可观测性问题？

A. 变量名是否足够短  
B. 上线后错误率、接口失败率、性能指标是否异常，以及异常时能不能定位  
C. 按钮颜色是否喜欢  
D. README 字数是否足够多

答案：B

解析：可观测性关注线上系统状态，帮助团队发现、定位和恢复问题。

### Q5

Azure DevOps、GitHub Actions、Jenkins 这类东西是什么？

A. DevOps 思想本身  
B. 支撑 DevOps 实践的工具或平台  
C. 浏览器 API  
D. 数据库协议

答案：B

解析：工具可以支撑 DevOps，但 DevOps 本身是交付、运行和反馈闭环的工程实践。

# 微前端

## 1. 它属于哪个知识板块？

```txt
前端工程化
→ 大型应用架构
→ 微前端
→ 路由 / 沙箱 / 通信 / 资源加载
```

==它**不是“多个 iframe 拼一起”**。微前端真正解决的是**多团队、多技术栈、巨型应用**怎么在**运行时**拆出独立开发、独立部署、独立运行的子应用。==

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

==**微前端**解决的是“**单体前端应用越来越大、多个团队互相阻塞**”的问题。==

当一个前端应用同时承载几十个业务模块、多个团队同时开发、构建一次半小时、上线要排队、技术栈想升级又不敢动旧代码时，单体方案的协作成本就压不住了。

==所以微前端的核心**不是技术框架**，而是把单体在**部署粒度、加载粒度、所有权粒度**上都拆开：每个子应用**自己的仓库、自己的发布、自己的运行时**，主应用只负责**路由分发、资源加载和最小通信**。==

### 2.2 核心流程

```txt
主应用启动
→ 注册子应用列表（名称、入口、激活规则）
→ 监听路由或路径变化
→ 命中激活规则后加载子应用入口
→ 在隔离环境里执行子应用代码
→ 渲染子应用 DOM 到指定容器
→ 子应用切换时卸载并清理
```

### 2.3 关键词清单

1. host / shell：主应用，负责路由分发、布局、登录态、子应用注册。
2. remote / sub-app：子应用，独立开发、独立部署，运行时被主应用加载。
3. activation rule：激活规则，例如根据路由前缀决定加载哪个子应用。
4. JS sandbox：JS 沙箱，隔离子应用对全局对象（`window`、原型链、定时器）的污染。
5. CSS isolation：样式隔离，常用方式包括 Shadow DOM、命名空间前缀、scoped css、动态作用域。
6. shared dependencies：共享依赖，避免每个子应用都重复打包 React / Vue。
7. communication：跨应用通信，常见方式是事件总线、共享 store、URL、自定义事件。
8. Module Federation：webpack 5 / Rspack / Rsbuild 提供的运行时模块共享能力，子应用以 remote 的方式被主应用按需加载。
9. qiankun：基于 single-spa 的国内常用方案，提供 HTML Entry、JS 沙箱、样式隔离等工程封装。
10. single-spa：微前端最早的开源方案，提供子应用注册、生命周期、路由分发的最小内核。
11. wujie：基于 Web Components + iframe 的方案，借助原生 Shadow DOM 和独立 window 做隔离。
12. lifecycle：子应用生命周期，常见钩子是 bootstrap、mount、unmount、update。
13. asset prefix / public path：子应用静态资源加载路径，错了就会 404 或加载到主应用的资源。

### 2.4 一句面试版

==微前端的核心是用**运行时拆分边界**，把巨型前端应用拆成独立开发、独立部署、独立运行的子应用，并通过 **JS 沙箱、样式隔离、共享依赖、跨应用通信** 把多个子应用拼成一个用户视角的整体应用，关键挑战是**隔离与共享之间的权衡**。==

### 2.5 最小 demo / 最小案例

#### Module Federation 最小骨架

主应用配置：

```ts
import { defineConfig } from '@rsbuild/core'
import { pluginModuleFederation } from '@module-federation/rsbuild-plugin'

export default defineConfig({
  plugins: [
    pluginModuleFederation({
      name: 'host',
      remotes: {
        sub: 'sub@http://localhost:3001/mf-manifest.json'
      },
      shared: ['react', 'react-dom']
    })
  ]
})
```

子应用配置：

```ts
export default defineConfig({
  plugins: [
    pluginModuleFederation({
      name: 'sub',
      exposes: {
        './App': './src/App.tsx'
      },
      shared: ['react', 'react-dom']
    })
  ]
})
```

主应用按需加载子应用导出的组件：

```tsx
const SubApp = lazy(() => import('sub/App'))
```

#### qiankun 最小骨架

主应用：

```ts
import { registerMicroApps, start } from 'qiankun'

registerMicroApps([
  {
    name: 'sub-app',
    entry: '//localhost:3001',
    container: '#subapp-container',
    activeRule: '/sub'
  }
])

start({ sandbox: { strictStyleIsolation: true } })
```

子应用导出生命周期：

```ts
export async function bootstrap() {}
export async function mount(props: { container: HTMLElement }) {
  render(props.container)
}
export async function unmount() {
  unmountApp()
}
```

### 2.6 常见问题怎么定位？

#### 子应用静态资源 404

```txt
检查 publicPath 是否在子应用运行时被正确改写
→ 检查主应用域名与子应用资源域名是否跨域
→ 检查 base、path 与部署目录是否一致
```

#### 样式互相污染

```txt
确认子应用是否启用了样式隔离（Shadow DOM 或命名空间）
→ 全局样式是否写在子应用而非主应用
→ 第三方 UI 组件库的全局变量是否冲突
```

#### 切换子应用后内存或事件残留

```txt
unmount 钩子是否真的清理了路由 / store / 事件监听
→ 子应用的全局变量是否被沙箱拦截
→ 是否有 setInterval、Resize/Scroll Listener 没解绑
```

#### 共享依赖版本错配

```txt
shared 配置中是否声明了 singleton
→ 主应用和子应用的依赖大版本是否兼容
→ 是否同时存在两份 React / Vue 在内存里
```

#### 主应用挂掉，子应用全死

```txt
主应用是否做了错误边界
→ 子应用加载失败时是否退化到独立访问 fallback
→ 是否在主应用里直接 throw 阻塞了其它子应用
```

### 2.7 三种主流方案怎么区分？

| 维度 | Module Federation | qiankun | wujie |
| --- | --- | --- | --- |
| 隔离粒度 | 同一 window，依赖打包级共享 | JS 沙箱 + 样式隔离，逻辑层隔离 | 基于 iframe + Web Components，window 级隔离 |
| 框架感知 | 不绑框架，但需配合构建工具 | 不绑框架，但接入有约定 | 不绑框架 |
| 通信方式 | 直接 import + 自定义协议 | 共享 store / 事件 / props | 通过桥接 API 通信 |
| 独立调试 | 远程加载 + 类型同步需要工程化 | 子应用本身可独立运行 | 子应用本身可独立运行 |
| 适用场景 | 同公司、同栈、需要细粒度模块共享 | 多技术栈、传统单体改造 | 强隔离、避免互相污染 |

实际选型不是“哪个最好”，而是**主应用诉求**（融合还是独立）、**隔离要求**（强 vs 弱）、**构建工具栈**、**子应用是否要独立部署**这四个维度的组合。

### 2.8 是否值得深入？

值得深入，尤其是参与中后台、SaaS 平台、低代码、巨型 ToB 系统时。优先顺序：

1. 先理解“为什么要在运行时拆分”，而不是直接学某个框架。
2. 再掌握激活规则、生命周期、子应用入口、资源加载路径这条主线。
3. 然后理解 JS 沙箱与样式隔离的实现思路，知道哪些行为会被拦、哪些不会。
4. 接着掌握共享依赖、跨应用通信、错误边界与降级。
5. 最后再深入 Module Federation 的运行时协议、SSR 兼容、灰度发布与版本管理。

优先看官方资料：Module Federation、qiankun、single-spa、wujie。

## 3. 选择题自测

### Q1

下面哪种说法更符合微前端的核心定位？

A. 把页面拆成多个 iframe
B. 把巨型前端应用按部署、加载、所有权拆成独立开发、独立部署、独立运行的子应用
C. 用更多 npm 包替代单体
D. 替代后端微服务

答案：B

解析：微前端的关键是运行时拆分边界与组织层面的所有权拆分。

### Q2

JS 沙箱主要解决什么问题？

A. 提升 LCP
B. 隔离子应用对全局对象、原型链、定时器等的污染，避免相互影响
C. 替代登录鉴权
D. 自动写测试

答案：B

解析：JS 沙箱让多个子应用可以共享同一个 window 又不会互相破坏全局状态。

### Q3

子应用静态资源 404 时，最该先检查什么？

A. 主应用定义的 CSS 全局变量和配色主题
B. publicPath / 部署目录 / base 是否一致
C. 服务端容器中的数据库索引配置是否正确
D. 项目根目录下的 README 配置文件是否包含相关截图

答案：B

解析：子应用资源加载路径错配是最常见的微前端坑。

### Q4

共享依赖配置不当最容易导致什么？

A. 主应用和子应用各打包一份 React，运行时同时存在两份导致状态错乱
B. CSS 文件变大
C. Git 冲突
D. README 出错

答案：A

解析：React、Vue 这类框架必须保持单实例，否则 hooks、context、响应式系统都会出问题。

### Q5

下面哪种行为更接近“治理良好的微前端”？

A. 子应用挂掉直接把主应用 throw 死
B. 子应用 unmount 时清理路由、store、事件、定时器，并提供加载失败的降级 UI
C. 主应用直接修改子应用的 DOM
D. 所有子应用共用一个全局 window 对象不做任何隔离

答案：B

解析：明确生命周期、清理资源、错误边界与降级，是微前端稳定运行的底线。

# 性能指标与度量

## 1. 它属于哪个知识板块？

```txt
前端性能优化
→ 性能指标与度量
→ Core Web Vitals / Lighthouse / RUM / Performance API
```

==这一站的目标**不是背指标名**，而是建立“**用户觉得慢**”和“**工程上能定位、能回归、能守住**”之间的桥。==

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

==**性能指标与度量**解决的是“**性能优化靠感觉，改了也不知道有没有变好**”的问题。==

没有指标时，团队经常会陷入争论：有人觉得页面慢，有人觉得本地很快；有人说包大，有人说接口慢；改了很多优化点，最后不知道真实用户有没有受益。

==所以性能度量的核心是把体验拆成**加载、响应、稳定、资源、运行时**几类指标，并区分**实验室数据**和**真实用户数据**。==

### 2.2 核心流程

```txt
确定关键页面和关键用户路径
→ 选择指标：LCP / INP / CLS / TTFB / FCP / bundle size
→ 本地用 DevTools / Lighthouse 复现
→ 线上用 RUM 采集真实用户数据
→ 按页面、设备、网络、版本分组
→ 找到最差的 75 分位或 95 分位
→ 优化后对比同口径数据
```

### 2.3 关键词清单

1. Core Web Vitals：核心体验指标，主线是 LCP、INP、CLS。
2. LCP：首屏主要内容多久出现。
3. INP：用户交互后页面多久给出下一次绘制反馈。
4. CLS：页面是否发生非预期布局偏移。
5. TTFB：从请求开始到收到 HTML 首字节的时间。
6. FCP：页面第一次出现文本、图片等内容的时间。
7. lab data：实验室数据，本地可控，适合定位问题。
8. field data / RUM：真实用户数据，来自真实设备和真实网络，适合判断真实体验。
9. percentile：分位数。看性能不要只看平均值，常看 p75、p95。
10. Performance API：浏览器提供的性能时间线 API，可以采集 navigation、resource、paint、event 等数据。

### 2.4 一句面试版

==性能指标与度量的核心是用 **LCP、INP、CLS** 表达用户体验，用 **DevTools / Lighthouse** 定位实验室问题，用 **RUM / Performance API** 采集真实用户数据，并按页面、设备、网络和版本做分组对比，避免“感觉优化”。==

### 2.5 最小 demo / 最小案例

用 `web-vitals` 采集核心指标：

```ts
import { onCLS, onINP, onLCP } from 'web-vitals'

function report(metric: unknown) {
  navigator.sendBeacon('/rum/web-vitals', JSON.stringify(metric))
}

onLCP(report)
onINP(report)
onCLS(report)
```

用 Performance API 看导航与资源：

```ts
const navigation = performance.getEntriesByType('navigation')[0]
const resources = performance.getEntriesByType('resource')

console.log(navigation)
console.table(resources.slice(0, 10))
```

### 2.6 实验室数据和真实用户数据怎么分工？

| 类型 | 代表工具 | 适合回答 |
| --- | --- | --- |
| 实验室数据 | DevTools、Lighthouse、本地压测 | 为什么慢、瓶颈在哪里、怎么复现 |
| 真实用户数据 | RUM、CrUX、线上埋点 | 真实用户是否变快、哪些页面/设备/地区最差 |
| 构建数据 | bundle analyzer、CI artifact | 这次发布是否让资源体积变大 |

### 2.7 常见误区

1. 只看 Lighthouse 分数，不看真实用户数据。
2. 只看平均值，不看 p75、p95。
3. 只看首页，不看真实高频业务路径。
4. 优化前后测试条件不一致。
5. 指标上报没有带版本、页面、设备、网络信息，导致无法定位。

## 3. 选择题自测

### Q1

性能优化为什么要先度量？

A. 因为没有指标就无法判断瓶颈和优化结果
B. 因为度量可以自动修复所有问题
C. 因为浏览器不能运行代码
D. 因为 Lighthouse 分数一定代表所有用户

答案：A

### Q2

真实用户数据最适合回答什么问题？

A. 真实用户在真实设备和网络下是否变快
B. 哪一种 CSS 选择器的命名更加符合 BEM 设计
C. 代码仓库中多分支提交流水线的命名规范
D. 本地开发机的 CPU 主频及物理配置是否满足编译要求

答案：A

### Q3

为什么常看 p75 而不是只看平均值？

A. p75 更能代表多数用户中偏慢的一批体验
B. p75 能自动压缩 JS
C. p75 不需要采集数据
D. 平均值永远错误

答案：A

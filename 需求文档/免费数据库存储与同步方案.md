# 免费数据库存储与云端同步方案

面向个人和分享朋友使用的前端学习系统，为解决多端数据同步、防数据丢失以及多用户数据隔离问题，特引入 Supabase 免费云端数据库作为存储服务，并与本地 IndexedDB 组成“离线优先、手动云同步”的混合架构。

---

## 一、 需求分析与目标

1. **多用户数据隔离**：
   - 系统支持多用户使用（主要为个人和分享给朋友）。
   - 每个用户需要有独立的账户体系（采用邮箱免密神奇链接 Magic Link 登录，安全高效，免记密码）。
   - 每个用户只能读写自己的数据（学习进度、错题本、刷题记录等），不能跨用户越权访问。

2. **离线优先与高可用性**：
   - 默认主存储依旧为浏览器本地的 IndexedDB，保证在无网、断网或云端不可用时，系统依然能够流畅使用。
   - 所有刷题记录、宠物状态等高频更新依然直接写入本地。

3. **手动云端同步**：
   - 提供一个用户可见的“云同步”控制面板。
   - 用户可以手动点击“立即同步”按钮，触发本地 IndexedDB 与云端 Supabase 数据库的双向合并。
   - 支持显示上一次成功同步的时间。

4. **安全与防刷**：
   - 数据库接口需要通过 RLS (Row Level Security，行级安全策略) 保证即使 API 密钥暴露在前端，也无法被恶意篡改其他用户的数据。

---

## 二、 技术选型：Supabase

- **数据库**：托管的 PostgreSQL (免费额度 500MB 存储，足够个人和朋友轻量使用多年)。
- **认证**：内置 Supabase Auth (月活 5 万免费用户额度)，采用邮箱免密神奇链接（Magic Link）进行免密鉴权。
- **客户端**：纯前端使用 `@supabase/supabase-js` 直连云端，无需开发与维护后端 API 服务。

## 五、 按需与增量双向同步算法设计

同步服务定义在 [syncService.js](file:///Users/neoyuan/Desktop/aoyi/个人中高级前端题库/学习系统/src/storage/syncService.js) 中，其核心机制如下：

1. **本地数据结构升级与脏数据追踪**：
   - 修改 `learnDb.js` 中的 `dbPut` 函数，向本地 IndexedDB 写入数据时，若传入的是普通对象，自动注入或更新本地毫秒级时间戳 `updatedAt = Date.now()`。
   - 增加脏标记机制：写入成功后，同时记录 `localStorage.setItem('local_db_dirty', 'true')` 标志本地产生未同步的更改。
   - **避空机制**：为 `dbPut` 引入 `isSync` 参数。若是由于从云端下载数据写入本地，则不触发脏标记，防止状态循环复脏。

2. **消灭错题时的软删除**：
   - 错题答对移出时，业务上从物理删除（`dbDelete`）改为带有被删除标记的 `dbPut(db, 'wrongBook', { qid, isDeleted: true, updatedAt: Date.now() })`。
   - `useLearningDb` 在初始化错题本缓存时，过滤掉带有 `isDeleted: true` 标记的数据。

3. **云端状态判定与快速通道（Version 控制）**：
   - 同步第一步先发起对云端 `user_sync_status.last_updated_at` 的极轻量单行查询，获得云端最后一次被更新的时间戳。
   - **快速判定退出**：如果云端的 `last_updated_at` 小于或等于本地的上一次同步时间 `last_cloud_sync_time`，且本地脏标记 `local_db_dirty === false`，则直接更新页面展示时间并宣告同步成功，不再向业务大表发送任何拉取请求，从而实现 0 业务表查询的数据库负载优化。

4. **增量拉取与双向合并**：
   - 若确实需要同步，调用 `getCloudData(userId, lastCloudSyncTime)` 时自动追加条件过滤：只获取在 `last_cloud_sync_time` 之后更新的记录，彻底避免了拉取历史上全量累积记录的性能和网络压力。
   - **单向只增历史（如答题历史 quiz_records）**：以毫秒时间戳 `time` 作为查重和过滤标准，两端互补在 `last_cloud_sync_time` 之后缺失的记录。
   - **可修改型业务表（进度、错题、刷题统计、宠物状态与事件）**：
     - 若仅本地存在该主键，且是在上次同步后更改的（`updatedAt > last_cloud_sync_time`），则增量上传至云端；仅云端存在，则增量拉取并写入本地。
     - 若两端均存在：对比本地 `updatedAt` 与云端 `updated_at` 转换为毫秒后的数值，新值覆盖旧值。

5. **上传回写与状态清除**：
   - 如果本次同步有本地增量数据成功上传（`uploadCount > 0`），则自动向云端 `user_sync_status` 写入最新更新的 `last_updated_at` 时间戳。
   - 同步成功后，重置本地脏标记为 `false`，并更新本地成功同步的对齐时间戳 `last_cloud_sync_time = Date.now()`。
   - 调用 `reloadLocalCaches` 重载 React 全局 State，使 UI 界面无感更新。

# 前端工程化

## 1. 陌生技术出现后的学习顺序

### 1.1 它解决什么问题？

==**前端工程化**解决的是前端项目从“**一个人本地能跑**”到“**团队长期稳定交付**”的问题。==

项目变大后，问题通常不在于会不会写页面，而在于依赖版本是否一致、代码风格是否统一、提交前能不能自动发现错误、构建产物是否稳定、不同环境配置是否清楚、发布失败后能不能定位和回滚。

==所以前端工程化**不是单个工具**，而是一套围绕**开发、检查、测试、构建、部署和监控**建立起来的标准流程。==

### 1.2 核心流程

```txt
创建项目结构
→ 管理依赖和脚本
→ 启动本地开发服务
→ 代码规范检查
→ 类型检查和自动化测试
→ 构建生产产物
→ CI 自动验证
→ 部署、监控和回滚
```

### 1.3 关键词清单

1. package manager：[[包管理器|包管理器]]，例如 npm、pnpm、Yarn，用来安装依赖并通过 lockfile 固定依赖版本。
2. scripts：`package.json` 里的命令入口，例如 `dev`、`lint`、`test`、`build`，让本地和 CI 使用同一套命令。
3. dev server：[[开发服务器|开发服务器]]，提供模块加载、热更新、代理等能力，让开发反馈更快。
4. bundler / build tool：[[构建工具|构建工具]]，例如 Vite、webpack、Rollup，用来处理依赖图、转换源码、拆分代码并输出生产产物。
5. lint / format：代码规范和格式化，例如 ESLint、Prettier，用来减少低级错误和风格争议。
6. type check：类型检查，例如 TypeScript 的 `tsc --noEmit`，用来在运行前发现类型层面的错误。
7. test：自动化测试，例如单元测试、组件测试、端到端测试，用来验证关键逻辑和用户路径。
8. environment：环境配置，例如 development、test、production，以及 `.env` 文件或构建时环境变量。
9. [[CI-CD|CI/CD]]：持续集成和持续部署，把安装依赖、检查、测试、构建和发布放到自动化流水线里执行。
10. performance optimization：[[前端性能优化|前端性能优化]]，围绕加载速度、交互响应、视觉稳定和资源体积持续优化用户体验。
11. observability：线上可观测性，例如 sourcemap、错误监控、性能指标，用来在发布后定位真实问题。

### 1.4 一句面试版

==前端工程化是用**工具、规范和自动化流程**，把前端项目从开发、检查、测试、构建到发布变成**可复用、可验证、可追踪**的体系，核心价值是**降低协作成本和交付风险**。==

### 1.5 最小 demo / 最小案例

一个最小前端工程化骨架，可以先从统一命令入口开始：

```json
{
  "scripts": {
    "dev": "vite",
    "lint": "eslint .",
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "build": "vite build",
    "preview": "vite preview"
  }
}
```

然后在 CI 中复用同一套命令：

```yaml
name: frontend-check

on:
  push:
    branches:
      - main
  pull_request:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-node@v6
        with:
          node-version: 22
          cache: npm
      - run: npm ci
      - run: npm run lint --if-present
      - run: npm run typecheck --if-present
      - run: npm run test --if-present
      - run: npm run build
```

这个案例不是完整工程化体系，但它已经包含主干：依赖安装可重复、命令入口统一、提交后自动检查、构建失败会阻断合并或发布。

### 1.6 是否值得深入？

值得深入，但不要一开始就陷进工具配置细节。优先顺序：

1. 先理解为什么需要统一依赖、脚本、代码规范、测试和构建。
2. 再学会读懂 `package.json`、lockfile、Vite 或 webpack 的基础配置。
3. 然后理解 CI 如何复用本地命令，避免“本地能跑，线上失败”。
4. 最后再深入 Monorepo、微前端、构建性能优化、发布策略、监控和回滚。

优先看官方资料：npm scripts、Vite build、webpack concepts、GitHub Actions workflows。

## 2. 选择题自测

### Q1

前端工程化最核心解决的问题是什么？

A. 让所有页面都必须使用同一个 UI 框架
B. 让前端项目的开发、检查、构建和发布流程标准化、自动化
C. 让 CSS 完全不需要维护
D. 让业务需求自动变成代码

答案：B

解析：前端工程化关注的是项目协作和交付流程的稳定性，不是绑定某个 UI 框架，也不能替代需求分析。

### Q2

`package-lock.json`、`pnpm-lock.yaml` 这类 lockfile 的主要作用是什么？

A. 记录单页应用（SPA）的前端路由拓扑路径
B. 固定依赖解析结果，让不同环境安装到尽量一致的依赖版本
C. 作为分布式版本管理系统来替代 Git 的核心版本控制能力
D. 自动分析并为项目中的所有模块生成组件使用文档

答案：B

解析：lockfile 用来记录依赖树的具体解析结果，降低“我这里能跑，你那里报错”的概率。

### Q3

为什么前端项目通常会把 `dev`、`lint`、`test`、`build` 写进 `package.json` 的 `scripts`？

A. 为了让命令入口统一，本地开发和 CI 都能复用
B. 为了隐藏源码
C. 为了让浏览器直接读取 `package.json`
D. 为了避免安装依赖

答案：A

解析：scripts 是工程命令的统一入口。团队成员和 CI 不需要记住底层工具细节，只要执行约定好的命令。

### Q4

下面关于构建工具的说法，哪一个更准确？

A. 构建工具只是把源文件复制到服务器
B. 构建工具只负责删除注释
C. 构建工具会围绕入口分析依赖，转换源码，并输出适合生产环境使用的产物
D. 构建工具只能用于后端项目

答案：C

解析：以 Vite、webpack 为例，构建工具会处理模块依赖、源码转换、资源处理、代码拆分和产物输出等问题。

### Q5

前端工程化中的一个常见误区是什么？

A. 先明确项目问题，再选择工具
B. 本地命令和 CI 命令尽量保持一致
C. 认为工具越多就越工程化
D. 用自动化检查提前发现问题

答案：C

解析：工程化的重点是降低协作成本和交付风险。工具只是手段，不能为了堆工具而增加维护成本。

# 传统软件开发工作流

## 1. 它属于哪个知识板块？

传统软件开发工作流属于：

```txt
软件工程基础
→ 软件开发生命周期（SDLC）
→ 需求 / 设计 / 开发 / 测试 / 发布 / 维护
→ 瀑布式流程 / 阶段门 / 交付物
```

==**传统软件开发工作流**解决的是“一个软件从想法到上线，再到后续维护，应该按什么阶段组织、谁负责什么、每个阶段产出什么”的问题。==

它不是某个工具，也不是某个岗位的专属流程。更准确地说：

==传统工作流通常把软件开发拆成一串相对清晰的阶段：**需求分析 → 方案设计 → 开发实现 → 测试验收 → 发布上线 → 运维维护**。每个阶段都有输入、输出和评审点。==

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

如果没有工作流，一个项目容易变成：

```txt
需求没说清就开始写
→ 写到一半发现方案不对
→ 测试阶段才发现核心逻辑理解错
→ 上线靠人工临时操作
→ 出问题没人知道该找谁
→ 修完 bug 又引入新问题
```

传统工作流的价值是把混乱拆开：

1. 先确认要解决什么问题。
2. 再设计怎么解决。
3. 然后写代码。
4. 再验证代码是否满足需求。
5. 最后发布给用户使用。
6. 上线后继续修 bug、处理反馈和迭代。

==传统工作流的核心不是“慢”，而是用**阶段、文档、评审和交付物**降低协作不确定性。它适合需求稳定、变更成本高、合规要求强、多人协作边界清楚的项目。==

## 3. 核心流程

最常见的传统流程可以画成这样：

```txt
需求收集
→ 需求分析
→ 技术方案 / 架构设计
→ 任务拆分与排期
→ 编码实现
→ 代码评审
→ 测试验证
→ 缺陷修复
→ 发布上线
→ 运维维护
→ 用户反馈进入下一轮需求
```

更简化的版本：

```txt
想清楚做什么
→ 设计怎么做
→ 把它做出来
→ 验证它对不对
→ 发给用户
→ 维护和改进
```

==这条链路就是 SDLC 的基本心智模型：软件不是只靠“写代码”完成的，而是经历从需求到维护的一整套生命周期。==

## 4. 每个阶段在做什么？

### 4.1 需求阶段

目标：弄清楚要解决什么业务问题。

常见参与者：

1. 产品经理。
2. 业务方。
3. 用户代表。
4. 研发负责人。
5. 测试负责人。

常见产出：

1. PRD / 需求文档。
2. 用户故事。
3. 业务流程图。
4. 验收标准。
5. 非功能需求，例如性能、安全、兼容性、可用性。

关键问题：

```txt
谁用？
解决什么问题？
成功标准是什么？
边界是什么？
哪些不做？
异常情况是什么？
```

### 4.2 设计阶段

目标：把需求翻译成可实现的技术方案。

常见产出：

1. 技术方案文档。
2. 架构图。
3. 数据模型。
4. 接口设计。
5. 页面流程。
6. 状态流转。
7. 风险点和降级方案。

前端项目里，这一阶段可能包含：

1. 路由怎么设计。
2. 组件怎么拆。
3. 状态放在哪里。
4. API 怎么调用。
5. 权限怎么控制。
6. 异常和 loading 怎么处理。
7. 是否影响性能、缓存、兼容性。

==设计阶段的重点不是写一堆漂亮文档，而是在动手前把**边界、依赖、风险和实现路径**想清楚。==

### 4.3 开发阶段

目标：按照需求和方案写出可运行的软件。

常见活动：

1. 创建分支。
2. 拆任务。
3. 编码。
4. 写单元测试。
5. 本地自测。
6. 提交代码。
7. 代码评审。
8. 合并到主分支或集成分支。

这一阶段的产出不是“代码越多越好”，而是：

```txt
可读
可测
符合设计
符合需求
能被团队维护
```

### 4.4 测试阶段

目标：确认软件是否满足需求，是否存在明显缺陷。

常见测试：

1. 单元测试。
2. 集成测试。
3. 系统测试。
4. 回归测试。
5. 验收测试。
6. 性能测试。
7. 安全测试。
8. 兼容性测试。

传统流程里，测试经常被放在开发之后；现代流程会把很多测试前移到开发阶段和 CI 流水线。

==测试阶段不是“找测试同学背锅”，而是用系统化验证降低上线风险。==

### 4.5 发布阶段

目标：把经过验证的软件交付到目标环境。

常见活动：

1. 打包构建。
2. 生成产物。
3. 配置环境变量。
4. 部署到测试 / 预发 / 生产环境。
5. 数据库变更。
6. 灰度或全量发布。
7. 发布检查。
8. 回滚准备。

在现代工程里，这一阶段通常会和 [[CI-CD|CI/CD]]、[[DevOps|DevOps]]、[[发布策略|发布策略]] 连接起来。

### 4.6 维护阶段

目标：软件上线后继续稳定运行和持续改进。

常见活动：

1. 修复线上 bug。
2. 处理用户反馈。
3. 监控性能和错误。
4. 优化体验。
5. 安全补丁。
6. 版本升级。
7. 技术债治理。
8. 下线废弃功能。

==维护不是开发结束后的杂活，而是软件生命周期的一部分。真正的软件会在上线后继续变化。==

## 5. 传统工作流和瀑布模型是什么关系？

很多人说“传统工作流”，通常是在说瀑布式流程。

瀑布模型的特点是：

```txt
一个阶段完成并评审通过
→ 再进入下一个阶段
→ 每个阶段依赖前一个阶段的产出
```

例如：

```txt
需求冻结
→ 设计评审
→ 开发完成
→ 测试完成
→ 发布上线
```

它的优点：

1. 阶段清楚。
2. 责任边界清楚。
3. 文档完整。
4. 管理和排期更容易。
5. 适合需求稳定、审批严格、合规要求强的项目。

它的问题：

1. 反馈晚，用户很晚才能看到真实产品。
2. 需求变更成本高。
3. 前期理解错，后期返工很重。
4. 开发、测试、运维容易形成墙。
5. 阶段交接多，信息容易丢失。

==瀑布式流程最大的风险是：把软件当成可以一次性想清楚、一次性交付正确的东西。但真实软件往往需要持续反馈和调整。==

## 6. 传统工作流和现代工作流的区别

现代敏捷、DevOps、CI/CD 并不是把传统流程全部推翻，而是改变它们的组织方式。

| 维度 | 传统工作流 | 现代工作流 |
| --- | --- | --- |
| 阶段关系 | 串行，前一阶段完成后进入下一阶段 | 迭代，多个阶段在小周期里反复发生 |
| 反馈时间 | 偏晚，测试和用户反馈常在后期 | 更早，PR、测试、灰度、监控都提供反馈 |
| 发布方式 | 大版本、低频发布 | 小批量、高频发布 |
| 文档 | 强调前期文档和审批 | 强调足够文档 + 持续沟通 |
| 测试 | 常在开发后集中测试 | 测试左移，自动化测试进 CI |
| 运维 | 上线后交给运维 | 开发、运维、平台一起对运行结果负责 |

可以这样理解：

```txt
传统工作流：把阶段分开管理
现代工作流：把阶段压缩进更短的反馈循环
```

==现代流程不是不要需求、设计、测试和发布，而是让这些活动更早、更小批量、更自动化、更可反馈。==

## 7. 程序员在每个阶段应该看什么？

### 7.1 需求阶段

不要只问“什么时候开发完”，要问：

1. 用户是谁？
2. 主流程是什么？
3. 边界条件是什么？
4. 验收标准是什么？
5. 哪些场景不做？
6. 是否有性能、安全、兼容性要求？

### 7.2 设计阶段

重点看：

1. 模块边界。
2. 数据流。
3. 状态流转。
4. 接口契约。
5. 错误处理。
6. 可测试性。
7. 发布和回滚风险。

### 7.3 开发阶段

重点看：

1. 代码是否符合设计。
2. 是否有必要的测试。
3. 是否影响已有功能。
4. 是否能被别人读懂。
5. 是否存在隐藏依赖和特殊环境假设。

### 7.4 测试阶段

重点看：

1. 主流程是否通过。
2. 异常流程是否覆盖。
3. 边界条件是否验证。
4. 回归范围是什么。
5. 是否有自动化测试补位。

### 7.5 发布阶段

重点看：

1. 发布产物是什么。
2. 环境变量是否正确。
3. 是否需要灰度。
4. 如何验证发布成功。
5. 出问题怎么回滚。

### 7.6 维护阶段

重点看：

1. 线上错误。
2. 性能指标。
3. 用户反馈。
4. 技术债。
5. 依赖升级。
6. 安全漏洞。

## 8. 最小 demo / 最小案例

假设你要做一个“用户资料编辑”功能，传统工作流会这样走：

```txt
需求：用户可以编辑昵称、头像、简介
→ 设计：确认表单字段、接口、校验规则、错误提示、权限
→ 开发：写页面、组件、接口调用、状态处理
→ 测试：验证必填、长度、头像上传失败、无权限、保存成功
→ 发布：构建产物，部署到测试，再灰度到生产
→ 维护：监控保存失败率，修复线上问题，处理用户反馈
```

每个阶段都有一个明确产出：

| 阶段 | 产出 |
| --- | --- |
| 需求 | PRD、验收标准 |
| 设计 | 技术方案、接口约定、页面流程 |
| 开发 | 代码、单测、自测记录 |
| 测试 | 测试用例、缺陷单、测试报告 |
| 发布 | 构建产物、发布记录、回滚方案 |
| 维护 | 线上问题记录、优化需求、补丁版本 |

这就是传统工作流的直观样子。

## 10. 一句面试版

==传统软件开发工作流通常指 SDLC / 瀑布式一类阶段化流程，它把软件从想法到上线拆成**需求分析、方案设计、开发实现、测试验证、发布上线、运维维护**等阶段，通过文档、评审和交付物管理协作风险；现代敏捷、DevOps 和 CI/CD 并不是删除这些阶段，而是把它们做得更小批量、更自动化、更早反馈。==

## 11. 是否值得深入？

值得。你不需要迷信传统流程，但必须理解它，因为公司里的需求评审、技术评审、提测、发布、回滚、线上维护，本质上都来自这套生命周期模型。

推荐深入顺序：

1. 先理解 SDLC 的六个基本阶段。
2. 再理解瀑布式流程为什么强调阶段门和交付物。
3. 然后理解敏捷为什么要缩短反馈周期。
4. 接着学 [[CI-CD|CI/CD]]，看自动化如何改造测试和发布。
5. 再学 [[DevOps|DevOps]]，看运行和反馈如何进入开发闭环。
6. 最后再看项目管理、需求分析、测试体系、发布治理和可观测性。

优先看官方资料：AWS SDLC、IBM SDLC、Atlassian Agile vs Waterfall。

## 12. 选择题自测

### Q1

传统软件开发工作流最核心解决的问题是什么？

A. 确保开发人员在整个研发生命周期中无需参与技术架构评审  
B. 把软件从想法到上线和维护拆成清晰阶段，明确每个阶段的输入、输出和责任  
C. 阻断自动化流水线，以便用纯人工操作替代全部安全测试  
D. 强制缩减产品环境的部署时间，规避所有的回归验证流程

答案：B

解析：传统工作流的价值是用阶段和交付物管理协作复杂度。

### Q2

下面哪个顺序最接近传统 SDLC 主线？

A. 发布 → 需求 → 测试 → 设计  
B. 需求 → 设计 → 开发 → 测试 → 发布 → 维护  
C. 测试 → 删除代码 → 需求  
D. 运维 → 不写代码 → 上线

答案：B

解析：不同模型细节不同，但需求、设计、开发、测试、发布、维护是最常见主线。

### Q3

瀑布式流程最典型的特点是什么？

A. 阶段顺序明确，通常前一阶段完成和评审后再进入下一阶段  
B. 每分钟自动上线一次  
C. 完全不需要文档  
D. 只适合写 CSS

答案：A

解析：瀑布模型强调线性、顺序和阶段交付物。

### Q4

传统工作流最大的常见风险是什么？

A. 反馈太早  
B. 用户很早就能验证产品  
C. 反馈太晚，需求或设计理解错误可能在后期才暴露，返工成本高  
D. 代码一定没有 bug

答案：C

解析：传统流程如果阶段过长、反馈过晚，很容易在测试或上线前才发现方向错了。

### Q5

现代 DevOps / CI/CD 和传统工作流的关系更准确的是？

A. 完全删除需求、设计、测试和发布  
B. 把这些阶段压缩进更短的反馈循环，并用自动化降低交付风险  
C. 只让运维写代码  
D. 不再需要维护

答案：B

解析：现代流程不是不要阶段，而是让阶段更小批量、更自动化、更可反馈。

# 中高级前端系统设计开放题：权限、实时协作与低代码

开放场景题考的不是“有没有标准答案”，而是你能不能把复杂业务拆成边界、数据、交互、性能、安全、发布和监控。

回答模板：

```txt
业务目标
→ 核心角色和流程
→ 数据模型
→ 前端模块边界
→ 异常和边界情况
→ 性能与稳定性
→ 安全与权限
→ 发布、监控、回滚
```

## 1. 权限系统：菜单、按钮、字段和数据权限怎么设计？

题目：

> 设计一个企业后台权限系统，要求支持菜单权限、按钮权限、字段权限和多租户数据权限。

回答骨架：

| 层级 | 前端负责 | 后端必须负责 |
| --- | --- | --- |
| 菜单权限 | 控制导航展示、路由可见性 | 返回用户可访问菜单 |
| 路由权限 | 拦截非法页面访问，提升体验 | 接口鉴权，不能只靠前端 |
| 按钮权限 | 控制操作入口展示和禁用 | 校验操作权限 |
| 字段权限 | 控制字段显示、脱敏、可编辑性 | 返回脱敏数据或字段策略 |
| 数据权限 | 展示范围提示 | 资源级鉴权、多租户隔离 |

关键判断：

- 前端权限不是安全边界，只是体验层。
- 权限数据要可缓存，但权限变化后要能失效。
- 权限粒度越细，配置、排查和性能成本越高。
- 多租户场景必须做资源级鉴权，不能只看角色名。

常见追问：

- 用户打开页面后权限被管理员收回怎么办？
- 按钮隐藏和按钮禁用怎么选？
- 前端如何避免权限配置散落在各组件里？
- 权限缓存怎么和登录态、租户切换联动？

## 2. 实时协作：多人编辑文档怎么设计？

题目：

> 设计一个多人协作文档或配置编辑器，多个用户同时编辑，需要看到在线状态、保存状态和冲突处理。

回答骨架：

```txt
连接层：WebSocket / SSE / WebRTC
  ↓
消息层：操作消息、心跳、重连、确认、去重
  ↓
协作模型：锁、版本号、OT 或 CRDT
  ↓
状态层：本地草稿、远端快照、保存状态
  ↓
体验层：光标、在线用户、冲突提示、离线恢复
```

方案取舍：

- 简单表单协作：字段级锁或版本号冲突检测即可。
- 富文本协作：需要 OT 或 CRDT 这类协作算法。
- 强实时互动：WebSocket 更合适。
- 服务端单向推送进度：SSE 可能更简单。

中高级回答要强调：

- 消息必须有 id、版本、时间、发送方和幂等处理。
- 断线重连后不能盲目覆盖本地草稿。
- 保存成功不等于所有协作者状态一致。
- 冲突策略要产品化：自动合并、提示用户选择、只读保护。

## 3. 低代码表单/页面搭建怎么设计？

题目：

> 设计一个低代码表单搭建平台，业务方可以拖拽字段、配置校验、发布页面。

回答骨架：

| 模块 | 设计重点 |
| --- | --- |
| Schema | 描述组件、布局、数据源、校验、联动、权限 |
| Renderer | 根据 schema 渲染页面，组件注册和沙箱边界 |
| Designer | 拖拽、选中、配置面板、撤销重做 |
| Runtime | 数据提交、校验、权限、错误回显 |
| Plugin | 组件扩展、物料市场、版本兼容 |
| Publish | 草稿、预览、审批、灰度、回滚 |

核心取舍：

- Schema 越灵活，运行时越复杂。
- 插件越开放，安全和兼容成本越高。
- 表达能力和可维护性要平衡，不要把平台做成“另一门没人会维护的语言”。

常见追问：

- 组件版本升级后旧页面怎么兼容？
- 动态表达式如何防 XSS？
- 表单联动如何避免循环依赖？
- schema 怎么做版本迁移？
- 预览和线上运行时如何保证一致？

## 4. 答题时的“中高级味道”

普通回答：

> 我会做菜单权限、按钮权限，然后用 WebSocket 做实时协作。

中高级回答：

> 我会先区分体验层权限和安全边界。前端控制菜单、按钮、字段展示，但后端必须做接口和资源级鉴权。权限配置要集中化，支持租户切换和权限变更后的缓存失效。实时协作会根据业务复杂度选择字段锁、版本冲突、OT 或 CRDT，而不是一上来只说 WebSocket。

差异在于：

- 有边界。
- 有取舍。
- 有异常。
- 有验证。
- 有长期维护成本。

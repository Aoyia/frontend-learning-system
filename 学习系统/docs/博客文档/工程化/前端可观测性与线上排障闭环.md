# 前端可观测性与线上排障闭环

中高级前端不只是“能写页面”，还要能回答：**线上客户出问题时，我如何知道、如何定位、如何止血、如何防止再次发生**。

可观测性解决的是“线上真实用户发生了什么我不知道”的问题。它不是埋点数量越多越好，而是建立从用户现象到证据链的排障系统。

## 1. 它解决什么问题？

典型客户问题：

- 用户说白屏，但本地复现不了。
- 某地区加载慢，但办公室网络正常。
- 新版本发布后错误率升高，但不知道是不是前端导致。
- 客户点击提交没反应，日志里只有后端成功。
- 线上 sourcemap 没有上传，错误栈全是压缩后的 `a.b.c`。

一句话：

> 前端可观测性要把错误、性能、用户行为、发布版本和请求链路串起来，让线上问题可以被发现、定位、归因和复盘。

## 2. 核心流程

```txt
用户发生问题
  ↓
采集错误 / 性能 / 行为 / 网络 / 版本
  ↓
聚合到监控平台
  ↓
按 release、页面、浏览器、地区、用户分组
  ↓
关联 traceId / requestId / sourcemap
  ↓
定位影响范围和根因
  ↓
灰度止血 / 回滚 / 修复
  ↓
补告警、测试、门禁和复盘
```

## 3. 关键指标与数据

| 数据 | 解决什么问题 |
| --- | --- |
| JS Error | 运行时异常、资源加载错误、Promise 未捕获 |
| RUM 性能 | LCP、INP、CLS、TTFB、资源耗时 |
| 用户行为 | 点击路径、页面停留、关键操作前后发生了什么 |
| Network | 请求状态、耗时、错误码、接口 traceId |
| Release | 哪个版本引入、灰度范围、回滚依据 |
| Sourcemap | 把压缩栈还原到源码位置 |
| Trace ID | 串联前端、网关、BFF、后端服务 |

## 4. 真实客户问题

客户反馈“新版本上线后订单页偶发白屏”。

排查路径：

1. 先看错误率是否从某个 release 开始升高。
2. 按页面过滤，只看订单页。
3. 按浏览器和设备聚合，确认是否集中在低版本 WebView。
4. 用 sourcemap 还原错误栈，定位到一个新引入的可选链编译产物兼容问题。
5. 查构建配置，发现目标浏览器范围没有覆盖客户设备。
6. 先回滚或关闭灰度，再修复 browserslist 和构建产物。
7. 补一个兼容性检查和灰度错误率阈值。

这个回答比“我会看控制台”更符合中高级要求，因为它包含影响范围、版本归因、源码定位、止血和防回退。

## 5. Source Map 不是可有可无

生产环境错误栈通常经过压缩和混淆：

```txt
TypeError: Cannot read properties of undefined
at a.b (index-abc123.js:1:20488)
```

没有 sourcemap，排查只能靠猜。

更稳定的做法：

- 构建产物生成 sourcemap。
- sourcemap 上传到监控平台或私有存储。
- 线上静态站点不直接公开敏感 sourcemap。
- release id 与构建产物、sourcemap、Git commit 绑定。
- 发布后验证 sourcemap 是否能正确还原栈。

## 6. Trace ID 如何串联前后端？

前端请求可以携带或透传请求标识：

```ts
const traceId = crypto.randomUUID()

fetch('/api/orders', {
  headers: {
    'x-trace-id': traceId,
  },
})
```

真实项目里 traceId 往往由网关或 BFF 生成，前端重点是：

- 请求失败时把 traceId 记录进错误上下文。
- 用户报障时能按 traceId 找到后端日志。
- 前端监控、网关日志、后端日志使用同一个链路标识。

## 7. 告警不是越多越好

有效告警要回答：

1. 谁受影响？
2. 影响多大？
3. 从哪个版本开始？
4. 是否正在扩大？
5. 有没有自动止血策略？

更合理的告警：

- 某 release 的 JS Error rate 超过阈值。
- 某核心页面 LCP P75 退化超过 20%。
- 支付提交接口 5xx 比例升高。
- 静态资源加载失败集中在某 CDN 节点。
- 灰度用户错误率明显高于稳定版本。

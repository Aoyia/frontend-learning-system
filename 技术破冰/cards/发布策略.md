# 发布策略

## 1. 它属于哪个知识板块？

```txt
前端工程化
→ CI/CD
→ 发布策略
→ 灰度 / 金丝雀 / 蓝绿 / 回滚
```

==它**不是“点一下按钮就上线”**。发布策略真正回答的是：**新版本怎么以最小代价暴露给最少用户、出问题怎么自动止损、回滚怎么真正回得去**。==

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

==**发布策略**解决的是“**上线就是最危险的时刻**”这个问题。==

CI 跑绿了不代表线上没事。真实生产里，问题往往在新版本接入真实流量、真实环境、真实数据后才暴露：脏数据、第三方接口不稳定、CDN 缓存未刷新、依赖配置遗漏、流量峰值、不同设备和地域差异。

==所以发布策略的核心**不是“怎么把代码搬到服务器”**，而是用**渐进式放量、健康度联动和可回滚的产物**，把**故障范围与时间窗口**控制到最小。==

### 2.2 核心流程

```txt
代码合入主干
→ CI 构建可回滚的产物或镜像
→ 选择发布策略（蓝绿 / 金丝雀 / 滚动 / 灰度）
→ 小流量验证（金丝雀）
→ 健康指标达标后逐步放量
→ 异常时自动暂停或回滚
→ 全量稳定后清理旧版本
```

### 2.3 关键词清单

1. release vs deploy：deploy 是把产物部署上去，release 是让用户**真正访问到**。两者解耦后才能做灰度。
2. blue-green deployment：蓝绿发布，准备两套完整环境（蓝、绿），切换流量来上线，回滚就是切回去。
3. canary release：金丝雀发布，先把新版本暴露给极小比例流量，观察指标后再放量。
4. rolling update：滚动发布，按批次替换实例，每批替换前后做健康检查，常见于 K8s。
5. progressive delivery：渐进式交付，灰度 + 健康度自动判断 + 自动放量或回滚的统称。
6. feature flag：特性开关，把“代码上线”和“功能开启”解耦，按用户 / 比例 / 环境控制。
7. traffic split：流量切分，常见维度有比例、用户 ID、地域、设备、白名单。
8. health check：健康检查，readiness（是否可接流量）和 liveness（是否还活着）通常分开。
9. SLI / SLO：服务等级指标和目标，例如错误率 < 0.5%、P95 延迟 < 300ms，是放量与回滚的判断依据。
10. rollback：回滚，重新部署上一个稳定产物或切回旧实例，要做到“**在任何放量阶段都能秒级回滚**”。
11. immutable artifact：不可变产物，构建一次产物，多个环境复用同一份，避免“环境之间又重新构建一次”。
12. asset hash + index html：前端发布特有，HTML 引用带 hash 的静态资源；旧 HTML 引用旧 chunk，新 HTML 引用新 chunk。
13. dynamic import 失败兜底：发布期间用户拿到旧 HTML 但请求新 chunk 失败，需要触发刷新或降级。

### 2.4 一句面试版

==发布策略的核心是用 **金丝雀 / 蓝绿 / 滚动** 等方式做**渐进式放量**，配合 **健康指标（SLI/SLO）、特性开关、可回滚的不可变产物**，让上线变成**可控暴露 + 自动止损 + 秒级回滚**的过程；前端发布还要额外处理 **静态资源缓存、旧 HTML 加载新 chunk 失败** 这类客户端长尾问题。==

### 2.5 最小 demo / 最小案例

#### 最小金丝雀 + 自动回滚的 GitHub Actions 骨架

```yaml
name: release

on:
  push:
    tags:
      - "v*"

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-node@v6
        with:
          node-version: 22
          cache: npm
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist

  canary:
    needs: build
    runs-on: ubuntu-latest
    environment: production-canary
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist
      - run: ./scripts/deploy.sh --target=canary --weight=5
      - run: ./scripts/check-slo.sh --window=10m --error-rate-max=0.5

  rollout:
    needs: canary
    runs-on: ubuntu-latest
    environment: production
    steps:
      - run: ./scripts/deploy.sh --target=stable --weight=100
```

骨架表达的是：

1. 构建一次，产物供 canary 与 stable 复用，保证“同一份产物”。
2. canary 阶段先放 5% 流量，跑一段时间内的 SLO 检查脚本。
3. 检查通过才进入全量放量；检查失败时这个 job 失败，可以挂回滚 hook。

#### Feature Flag 把发布与开启解耦

```ts
import { isEnabled } from '@/feature-flags'

if (isEnabled('new-pricing-page', { userId, region })) {
  return <NewPricingPage />
}
return <LegacyPricingPage />
```

特性开关的价值：

```txt
代码上线 ≠ 功能开启
→ 出问题时不用改代码、不用回滚部署
→ 直接关闭开关
→ 还能按用户、地域、比例细分放量
```

#### 前端版本切换的客户端坑

```txt
新版本部署
→ HTML 文件被 CDN 刷新到 v2
→ 静态资源（带 hash）保留 v1 与 v2
→ 老用户浏览器缓存的 v1 HTML 仍然引用 v1 chunk
→ 用户切换路由触发动态 import
→ 老 HTML 引用的 v1 chunk 已被清理 → 加载失败
```

兜底做法：

```ts
window.addEventListener('error', e => {
  const target = e.target as HTMLElement | null
  if (target && (target.tagName === 'SCRIPT' || target.tagName === 'LINK')) {
    location.reload()
  }
})
```

或在框架路由层捕获 chunk 加载失败时，提示用户刷新或自动 reload。

### 2.6 几种策略怎么区分？

| 策略 | 核心做法 | 优点 | 代价 |
| --- | --- | --- | --- |
| 蓝绿 blue-green | 准备两套完整环境，切流量来上线 | 切换原子，回滚就是切回去 | 需要双倍资源；数据库变更要双向兼容 |
| 金丝雀 canary | 先把新版本暴露给极小比例流量 | 故障半径小，能用真实流量验证 | 需要精细的流量切分与健康度判断 |
| 滚动 rolling | 按批次替换实例，每批做健康检查 | 资源占用低，K8s 默认 | 出问题时可能多个版本并存中 |
| 灰度（业务维度） | 基于 feature flag / 用户白名单放量 | 与代码部署解耦，可按业务维度切 | 需要建设特性开关基建 |
| 影子流量 shadow | 把生产流量复制一份打到新版本，不返回结果 | 真流量验证不影响用户 | 副作用需谨慎，写流量难做影子 |

实际线上很少只用一种。常见组合是：**滚动部署底座 + 金丝雀放量 + Feature Flag 控开启**，配合 SLO 自动判断是否继续放量。

### 2.7 回滚要回得去的几条硬规矩

==很多“能回滚”的项目其实回不去，因为忘了下面几条。==

1. **产物不可变**：回滚意味着重新部署一个**已经存在的旧产物**，不是“拿旧 commit 重新构建一次”。
2. **数据库迁移要前向兼容**：新版本的 DDL 不能让旧版本启动失败，破坏性变更要拆成多步上线。
3. **配置随版本走**：配置和代码必须一起版本化，不要回滚了代码、配置还停留在新版本。
4. **接口契约要兼容**：新增字段可选、删除字段要给过渡期，否则前后端版本错位会立刻炸。
5. **缓存要可清理**：CDN、Service Worker、客户端本地存储都可能让回滚“看不到效果”。
6. **回滚脚本要演练**：从未跑过的回滚脚本等于没有回滚。

### 2.8 是否值得深入？

值得深入，特别是参与中后台、SaaS、ToB 系统、有真实生产流量的场景。优先顺序：

1. 先理解 release / deploy 解耦，与“产物不可变”这两件事。
2. 再掌握蓝绿 / 金丝雀 / 滚动的边界与代价。
3. 然后引入 Feature Flag，把代码上线与功能开启解耦。
4. 接着把放量与 SLO 联动，实现自动暂停与自动回滚。
5. 最后处理前端发布特有的客户端缓存与 chunk 加载失败长尾问题。

优先看官方资料：Kubernetes Deployments、Argo Rollouts、LaunchDarkly / OpenFeature 文档、Google SRE Book 关于 release engineering 的章节。

## 3. 选择题自测

### Q1

下面哪种说法更准确？

A. release 和 deploy 是同义词
B. deploy 是把产物部署上去，release 是让用户真正访问到，两者解耦才能做灰度
C. CI 跑绿了就等于发布完成
D. 发布只关心后端

答案：B

解析：解耦 deploy 与 release 是渐进式发布的前提。

### Q2

金丝雀发布最核心的价值是什么？

A. 节省服务器
B. 用极小比例真实流量先验证新版本，把故障半径限制到最小
C. 替代单元测试
D. 让代码更短

答案：B

解析：小比例真实流量比任何预发环境都更接近线上真实情况。

### Q3

下列哪个是“真正能回滚”的硬条件？

A. 产物不可变，旧产物随时可以重新部署
B. README 里写一句“支持回滚”
C. 把所有日志删掉
D. 不留 sourcemap

答案：A

解析：回滚的本质是切回某一份**已经存在并验证过**的产物。

### Q4

Feature Flag 的核心价值是什么？

A. 替代部署
B. 把“代码上线”和“功能开启”解耦，按用户、比例、地域控制功能可见性，并能在不回滚部署的情况下关闭功能
C. 让 CSS 更小
D. 替代 lint

答案：B

解析：Feature Flag 让新功能上线后仍可被快速控制，是降低发布风险的重要手段。

### Q5

前端在发布期间最常见的客户端问题是什么？

A. 老用户浏览器缓存的旧 HTML 引用了已被清理的旧 chunk，导致动态 import 失败
B. CSS 自动重启服务器
C. JS 自动改 lockfile
D. README 自动重写

答案：A

解析：前端的“客户端缓存 + 动态导入”特性带来这种长尾问题，需要通过监听加载错误并刷新页面或提示用户来兜底。

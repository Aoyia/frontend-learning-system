# 前端性能优化

## 1. 陌生技术出现后的学习顺序

### 1.1 它解决什么问题？

==**前端性能优化**解决的是“页面能不能**更快出现、更快响应、更稳定显示**”的问题。==

用户感受到的慢，不只是一句“接口慢”或“包太大”。它可能发生在 DNS、TCP、TLS、服务端响应、HTML 解析、CSS 阻塞、图片加载、JavaScript 执行、框架渲染、布局计算、绘制合成、缓存策略等很多环节。

==所以性能优化**不能靠感觉**。正确顺序是**先度量，再定位瓶颈，最后针对瓶颈做优化，并验证优化是否真的生效**。==

### 1.2 核心流程

```txt
明确优化目标
→ 采集指标
→ 区分实验室数据和真实用户数据
→ 定位瓶颈环节
→ 按加载 / 交互 / 稳定性分类优化
→ 对比优化前后数据
→ 建立监控和性能预算
```

### 1.3 关键词清单

1. Core Web Vitals：核心用户体验指标，当前主线是 LCP、INP、CLS。
2. LCP：Largest Contentful Paint，最大内容绘制，关注首屏主要内容多久出现，建议做到 2.5 秒以内。
3. INP：Interaction to Next Paint，交互到下一次绘制，关注用户点击、输入后页面多久响应，建议做到 200ms 以内。
4. CLS：Cumulative Layout Shift，累计布局偏移，关注页面是否乱跳，建议做到 0.1 以内。
5. TTFB：Time To First Byte，首字节时间，反映从请求到收到 HTML 第一个字节的耗时。
6. FCP：First Contentful Paint，首次内容绘制，反映页面第一次出现文本、图片等内容的时间。
7. lab data：实验室数据，例如本地 DevTools、Lighthouse，适合开发阶段复现和定位问题。
8. field data / RUM：真实用户数据，来自真实用户设备、网络和页面访问，适合判断线上真实体验。
9. [[性能指标与度量|性能指标与度量]]：把“快不快”拆成可采集、可比较、可追踪的指标体系。
10. [[LCP 与首屏加载|LCP 与首屏加载]]：围绕首屏最大内容元素优化 HTML、CSS、图片、字体、接口和渲染阻塞。
11. [[INP 与交互响应|INP 与交互响应]]：围绕用户输入后的下一次绘制，优化长任务、事件回调、渲染范围和调度。
12. [[CLS 与视觉稳定|CLS 与视觉稳定]]：围绕布局偏移，处理图片尺寸、广告位、字体替换、异步插入内容。
13. [[资源加载与缓存策略|资源加载与缓存策略]]：围绕网络请求、资源优先级、压缩、缓存、CDN 和预加载做优化。
14. [[渲染流程与渲染性能|渲染流程与渲染性能]]：理解 HTML、CSSOM、layout、paint、composite，避免重排重绘和布局抖动。
15. [[性能监控与性能预算|性能监控与性能预算]]：把性能从一次性优化变成持续监控和发布门禁。
16. [[Chrome Memory 与 Performance Monitor|Chrome Memory / Performance Monitor]]：Chrome DevTools 里的性能诊断工具，用来观察运行时指标和定位内存问题。

### 1.4 一句面试版

==前端性能优化**不是盲目压缩文件**，而是围绕 **LCP、INP、CLS** 等用户体验指标，先用 **DevTools、Lighthouse 或真实用户监控**定位瓶颈，再从网络传输、资源体积、渲染路径、JavaScript 执行和缓存策略等环节逐步优化，并用**数据验证结果**。==

### 1.5 最小 demo / 最小案例

最小性能优化不要先改代码，先上报核心指标：

```ts
import { onCLS, onINP, onLCP } from 'web-vitals'

function sendToAnalytics(metric: unknown) {
  const body = JSON.stringify(metric)

  if (navigator.sendBeacon) {
    navigator.sendBeacon('/analytics/web-vitals', body)
    return
  }

  fetch('/analytics/web-vitals', {
    method: 'POST',
    body,
    keepalive: true
  })
}

onLCP(sendToAnalytics)
onINP(sendToAnalytics)
onCLS(sendToAnalytics)
```

然后按指标拆问题：

```txt
LCP 差
→ 看首屏最大元素是什么
→ 看 HTML、CSS、图片、字体、接口哪个阻塞它
→ 优先优化服务端响应、关键资源加载和首屏图片

INP 差
→ 打开 Performance 面板录制交互
→ 找主线程长任务
→ 拆分重计算、减少同步 JS、降低渲染范围

CLS 差
→ 找发生偏移的元素
→ 给图片、广告、异步内容预留尺寸
→ 避免在已有内容上方突然插入 DOM
```

### 1.6 知识地图

```txt
前端性能优化
→ 性能指标与度量：先知道慢在哪里
→ LCP 与首屏加载：让主要内容更快出现
→ INP 与交互响应：让点击、输入、拖拽更快反馈
→ CLS 与视觉稳定：让页面不乱跳
→ 资源加载与缓存策略：让资源更少、更小、更早、更可缓存
→ 渲染流程与渲染性能：让浏览器少做无用布局、绘制、合成
→ 性能监控与性能预算：让优化结果持续可见、可守住
```

### 1.7 常见优化方向

#### 加载速度

1. 减少首屏关键资源数量，避免无关 JS、CSS 抢占首屏带宽。
2. 图片使用合适尺寸和格式，首屏关键图优先加载，非首屏图片懒加载。
3. 使用代码分割，把路由级、弹窗级、低频功能拆成异步 chunk。
4. 开启 gzip 或 brotli 压缩，静态资源使用带 hash 的长缓存。
5. 服务端优化 TTFB，必要时使用 CDN、SSR、SSG 或接口缓存。

#### 交互响应

1. 避免一次性执行大段同步 JavaScript。
2. 拆分长任务，把非关键计算延后或放到 Web Worker。
3. 列表和表格做虚拟滚动，避免一次渲染大量 DOM。
4. 输入、拖拽、滚动等高频事件要节流、防抖或减少回调工作量。
5. 框架层减少无效渲染，控制组件更新范围。

#### 视觉稳定

1. 图片、视频、广告位、异步组件提前预留宽高。
2. 字体加载避免大幅度替换造成跳动。
3. 不在已有内容上方突然插入横幅、提示条或接口返回内容。
4. 骨架屏和占位元素要接近最终内容尺寸。

### 1.8 固定排查路径

```txt
先看真实用户数据
→ 再用实验室工具复现
→ 找到具体页面、设备、网络、用户路径
→ 按 LCP / INP / CLS 分类
→ 打开 DevTools 对应面板定位瓶颈
→ 改一个点，测一次
→ 上线后继续看 RUM 和性能预算
```

常见判断：

1. LCP 差，先看首屏最大元素、HTML TTFB、CSS 阻塞、LCP 图片、字体和客户端渲染链路。
2. INP 差，先看交互对应的事件回调、主线程长任务、组件重渲染范围和同步计算。
3. CLS 差，先看图片、广告、iframe、字体、异步内容是否预留空间。
4. Lighthouse 分数差但真实用户数据好，优先定位实验室建议是否适合你的真实业务场景。
5. 本地很快但线上慢，优先比较真实设备、弱网、缓存状态、CDN、接口、第三方脚本。

### 1.9 是否值得深入？

值得深入，但要按“指标 → 工具 → 原理 → 手段”走：

1. 先掌握 LCP、INP、CLS 分别代表什么用户感受。
2. 再学会用 Chrome DevTools Performance 面板看主线程、网络瀑布和布局偏移。
3. 然后理解浏览器渲染流程：HTML 解析、CSSOM、布局、绘制、合成。
4. 接着看构建产物：bundle 体积、chunk 拆分、重复依赖、资源压缩。
5. 最后建立线上监控和性能预算，避免优化后又退化。

优先看官方资料：Web Vitals、Chrome DevTools Performance、MDN Performance API、Lighthouse。

## 2. 选择题自测

### Q1

前端性能优化最正确的起手式是什么？

A. 先把所有代码重写一遍
B. 先采集指标，定位瓶颈，再针对问题优化
C. 先把所有图片删掉
D. 只看自己的电脑是否流畅

答案：B

解析：性能优化必须基于数据。没有指标和瓶颈定位，很容易优化错方向。

### Q2

LCP 主要反映什么体验？

A. 首屏主要内容多久出现
B. 用户点击后多久响应
C. 页面是否发生布局跳动
D. 代码有没有类型错误

答案：A

解析：LCP 关注最大内容元素的绘制时间，通常和首屏主要内容加载体验强相关。

### Q3

INP 差通常应该优先检查什么？

A. 主线程是否有长任务阻塞交互响应
B. CSS 文件名是否好看
C. Git 分支名是否规范
D. README 是否有目录

答案：A

解析：INP 关注交互响应，长任务、重计算、同步 JS 和过大的渲染范围都会拖慢它。

### Q4

CLS 差最常见的原因之一是什么？

A. 图片、广告或异步内容没有预留空间，加载后把页面挤开
B. 使用了 TypeScript
C. 使用了 Git
D. 使用了 npm scripts

答案：A

解析：布局偏移通常来自元素尺寸不确定、内容后插入、字体替换等问题。

### Q5

为什么只看 Lighthouse 分数不够？

A. Lighthouse 只能在实验室环境下模拟一次或几次访问，不能完全代表所有真实用户
B. Lighthouse 不会运行在浏览器里
C. Lighthouse 只能测试后端接口
D. Lighthouse 会自动修复所有问题

答案：A

解析：实验室数据适合定位问题，真实用户数据更适合判断线上整体体验。两者要结合看。

# 构建工具

## 1. 陌生技术出现后的学习顺序

### 1.1 它解决什么问题？

==**构建工具**解决的是“源码不能直接**稳定交付给生产环境**”的问题。==

真实项目里，代码通常会包含 TypeScript、JSX、Vue SFC、CSS 预处理器、图片字体、第三方依赖、环境变量、按需加载和浏览器兼容处理。浏览器最终需要的是可加载、体积可控、缓存友好、能在线上稳定运行的 HTML、CSS、JS 和静态资源。

==所以构建工具的核心价值**不是“把代码压缩一下”**，而是围绕入口**分析依赖图**，把各种源码和资源**转换、打包、拆分、优化**成适合开发和生产的产物。==

### 1.2 核心流程

```txt
读取入口
→ 解析模块依赖图
→ 解析和转换不同类型文件
→ 调用插件或 loader 扩展能力
→ Tree shaking 删除无用代码
→ 代码拆分和懒加载
→ 压缩、哈希命名和 sourcemap
→ 输出 dist 产物
```

开发环境还会多一步：

```txt
启动 dev server
→ 按需编译模块
→ 浏览器请求模块
→ 文件变化触发 HMR
→ 页面局部更新
```

### 1.3 关键词清单

1. entry：入口文件，构建工具从这里开始追踪依赖图，例如 `src/main.ts` 或 `index.html`。
2. module graph：模块依赖图，记录一个模块依赖了哪些模块和资源。
3. loader / transform：文件转换，把 TypeScript、JSX、Vue SFC、Less、图片等转换成构建工具能继续处理的形式。
4. plugin：插件机制，用来扩展构建流程，例如注入环境变量、生成 HTML、压缩资源、拷贝静态文件。
5. bundle：打包后的产物，把多个模块组合成浏览器或运行时能加载的文件。
6. chunk：拆分后的代码块，常见于路由懒加载、公共依赖拆分和动态导入。
7. tree shaking：基于 ESM 静态结构删除未使用代码，减少最终产物体积。
8. minify：压缩代码，删除空白、缩短变量名、简化表达式。
9. sourcemap：源码映射，让线上压缩后的报错能定位回原始源码。
10. dev server / HMR：开发服务器和热更新，让开发时不用每次完整重新构建。

### 1.4 一句面试版

==构建工具会从入口出发**分析模块依赖图**，通过 **loader、transform 和 plugin** 处理不同类型的源码和资源，最后输出经过**拆分、压缩、缓存优化**的生产产物；开发环境下还会提供 **dev server 和 HMR** 来提升反馈速度。==

### 1.5 最小 demo / 最小案例

以 Vite 项目为例，最小命令通常是：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
```

最小配置可以先只理解这几个点：

```ts
import { defineConfig } from 'vite'

export default defineConfig({
  base: '/',
  build: {
    outDir: 'dist',
    sourcemap: true
  }
})
```

这段配置表达的是：

1. `dev` 负责本地开发服务。
2. `build` 负责生产构建。
3. `preview` 用来本地预览生产产物。
4. `outDir` 决定产物输出目录。
5. `sourcemap` 决定是否生成源码映射，方便排查线上问题。

### 1.6 常见工具怎么区分？

1. [[Vite|Vite]]：现代前端应用开发工具，重视开发启动速度和开箱即用体验。当前 Vite 官方构建文档中，生产构建配置重点已经转向 `build.rolldownOptions`。
2. webpack：经典全能力打包器，核心概念是 entry、output、loader、plugin、mode，生态成熟，适合复杂历史项目和高度定制场景。
3. Rollup：更偏向 JavaScript 库打包，擅长基于 ESM 做 tree shaking，并输出 ES modules、CommonJS、UMD 等多种格式。
4. esbuild：用 Go 编写的高速构建工具，常用于转译、压缩和预打包，也可以直接做 bundling。
5. Rolldown：Rust 写的现代 bundler，目标是兼容 Rollup 生态并提升性能，已经进入 Vite 新版本构建链路。

### 1.7 是否值得深入？

值得深入，但不要从配置项背诵开始。优先顺序：

1. 先理解入口、依赖图、转换、插件、产物这条主线。
2. 再理解开发构建和生产构建为什么不一样。
3. 然后学会读构建产物：文件名哈希、chunk、vendor、sourcemap、assets。
4. 接着掌握常见问题：路径 base 错误、缓存没更新、chunk 加载失败、依赖重复打包、构建太慢。
5. 如果看到内部包、组件库、SDK 或 UMD 文件，再进入 [[内部 NPM 包与库模式构建|内部 NPM 包与库模式构建]]。
6. 最后再深入 loader/plugin 原理、tree shaking 边界、代码拆分策略、Monorepo 构建缓存。

优先看官方资料：Vite build、webpack concepts、Rollup introduction、esbuild API。

## 2. 选择题自测

### Q1

构建工具最核心解决的问题是什么？

A. 让所有代码只能写成 JavaScript
B. 把源码和资源转换、打包、优化成适合开发或生产环境使用的产物
C. 自动替产品经理写需求
D. 禁止使用第三方依赖

答案：B

解析：构建工具围绕源码到产物的转换流程工作，不负责替代需求分析，也不会限制只能写原生 JavaScript。

### Q2

entry 的作用是什么？

A. 表示构建工具开始分析依赖图的入口
B. 表示数据库连接地址
C. 表示浏览器缓存时间
D. 表示测试覆盖率

答案：A

解析：构建工具从入口开始递归分析模块之间的依赖关系，然后再做转换、打包和输出。

### Q3

loader、transform、plugin 这类机制的共同价值是什么？

A. 扩展构建工具能力，处理不同类型文件或介入构建流程
B. 让项目不能运行
C. 替代 Git
D. 删除所有业务代码

答案：A

解析：不同工具叫法不完全一样，但它们都在解决“构建流程如何扩展”的问题。

### Q4

Tree shaking 依赖最多的模块特性是什么？

A. CommonJS 的动态 `require`
B. ESM 的静态 `import` / `export` 结构
C. CSS 的类名
D. HTML 注释

答案：B

解析：ESM 的静态结构更容易在构建阶段分析哪些导出没有被使用，因此更适合做 tree shaking。

### Q5

下面哪种情况最适合先检查构建配置里的 `base` 或 public path？

A. 页面标题写错
B. 线上打开页面后 JS、CSS 或图片路径 404
C. 本地变量命名不喜欢
D. README 缺少截图

答案：B

解析：静态资源路径错误经常和部署路径、`base`、public path、CDN 前缀有关。

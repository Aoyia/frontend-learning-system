## 模块一：AI 协作规范与编码 SOP

在实际开发场景中，单纯把 AI 当作“代码生成打字员”会导致生成的代码风格迥异、系统架构被破坏，维护成本反而激增。行业领先团队普遍转向了**“契约驱动开发”**与**“施工队长”模式**。

### 1. 从“打字员”向“施工队长”模式转型
*   **痛点**：开发者直接向 AI 提问要求写一段代码，导致代码缺乏规范约束，难以维护。
*   **解决方案**：人类的核心职责是“决策、审批、把关架构”，AI 负责具体的代码开发与测试。人类以架构蓝图和研发契约驱动 AI。
*   **引用源**：[腾讯技术工程 - 《当整个团队开始0-Coding：一份万字 AI Native 研发实战手册》](https://zhuanlan.zhihu.com/p/2021966760346231991)

### 2. OpenSpec 研发契约规范
通过结构化的文档契约，封堵自然语言的多义性鸿沟，使 AI 能够准确理解并批量执行任务：
*   **需求定义（`proposal.md`）**：明确本次变更的背景、业务目标及“为什么做”。
*   **架构决策（`design.md`）**：定义模块设计、接口约定与关键技术决策。
*   **任务分解清单（`tasks.md`）**：由 AI 自动生成带复选框的施工清单，原则上控制在 15 项以内，避免 AI 因上下文过长而产生幻觉。
*   **规范增量（`specs/`）**：保存本次变更对系统原有规范的修改，作为项目活的知识库。
*   **引用源**：[腾讯技术工程 - 《当整个团队开始0-Coding：一份万字 AI Native 研发实战手册》](https://zhuanlan.zhihu.com/p/2021966760346231991)

## 模块二：AI 自动门禁（CI/CD 集成与 AI Code Review）

为了保质保量地干活，不能单靠开发者的自觉，必须在 CI/CD 流水线上构建自动化“门禁”，通过 AI 在人工审查之前拦截低级问题。

### 1. 分层代码审查（AI + 人工 CR）
*   **定位**：AI 门禁不以“完全替代人工”为目的。
*   **规范**：AI 负责在 CI 阶段提前拦截明显的低级 Bug（如空指针异常、未捕获的异常分支、硬编码敏感信息、类型定义 Error），让人工 Reviewer 把精力集中在业务架构和深层设计上。
*   **引用源**：[掘金 - 《实战：搭建 AI Code Review 自动化流水线》](https://juejin.cn/post/7646348958055628850)

### 2. 自动化 AI Code Review 流水线架构
行业推荐的轻量级落地架构如下：
*   **AI Review 后端服务**：基于 Python Flask 搭建。服务启动时从中央 Git 仓库同步团队的最新审查规范，接收代码 Diff 后结合“规范审查 + 专家审查”的 System Prompt 交给 LLM。LLM 会对代码进行 100 分制打分，并返回结构化的 JSON 格式报告。
*   **GitLab CI/CD 流水线集成**：在 `.gitlab-ci.yml` 中配置 `ai-review` 作业，配置触发条件为 `merge_request_event`。
*   **评论自动回写**：流水线通过 Python 脚本解析 AI 返回的 JSON，利用 GitLab PAT 令牌调用 `/merge_requests/:mr_iid/notes` 接口，自动在对应的 MR/PR 上贴出精简格式化的 Markdown 审查建议。
*   **引用源**：[掘金 - 《实战：搭建 AI Code Review 自动化流水线》](https://juejin.cn/post/7646348958055628850)

### 3. CI/CD 落地避坑实践
*   **GitLab Token 权限**：访问令牌必须被分配 `api` 级别权限，若仅分配 `read_api` 会导致评论写入接口报 401 错误。
*   **命令行 JSON 传输**：非 TTY 的 CI Runner 执行时，超长 payload 的 JSON 易发生转义乱码。执行 curl 时应使用 stdin 管道传入（如 `@-`）确保数据完整。
*   **引用源**：[掘金 - 《实战：搭建 AI Code Review 自动化流水线》](https://juejin.cn/post/7646348958055628850)

## 模块三：团队落地 AI 辅助研发的 SOP 与工作流

团队级落地 AI 工具（如 Claude Code, Cursor 等）时，需要制定明确的行为护栏和协同工作流，以防范影子 AI 与幻觉问题。

### 1. 三阶段对话模型 SOP
自然语言的多义性往往导致 AI 生成大量冗余或错误的代码。建议采用以下对话 SOP：
*   **阶段一（需求定义）**：使用“用户故事 + 验收标准（Acceptance Criteria）”结构化需求，明确系统边界。
*   **阶段二（边界明确）**：严格区分“必须遵守的技术约束（如分层架构规范、基类继承、安全日志）”和“仅供参考的成熟代码”。
*   **阶段三（迭代反馈）**：小步快跑。AI 生成前必须主动暂停（如表结构设计完毕后先由人核对，确认后 AI 再生成对应的 Service 和 Controller）。
*   **引用源**：[得物技术 - 《从Claude Code实践到团队协作的优化思考｜得物技术》](https://juejin.cn/post/7599828354514567208)

### 2. 团队开发规范文件（如 `CLAUDE.md`）设计艺术
*   **痛点**：规则文件写得过于臃肿、指令繁杂，导致 AI 记不住或者上下文开销过大。
*   **解决方案**：建议把核心规则文件控制在 200 字以内。它应该充当“高频错误护栏”，而不是“开发手册”。例如仅记录最常错的规范（禁止使用某类 API、非空校验约定），而具体的业务领域知识通过文档路径指引 AI 在需要时动态读取。
*   **引用源**：[得物技术 - 《从Claude Code实践到团队协作的优化思考｜得物技术》](https://juejin.cn/post/7599828354514567208)

### 3. 多 Agent（子代理）协同的“中间产物驱动”模式
*   **痛点**：多 Agent 协作时，如果 Agent 之间直接进行复杂的对话通信，容易导致状态不一致和信息风暴。
*   **解决方案**：多 Agent 围绕同一个“中间产物”（如技术方案设计文档）进行增量更新与消费。架构师 Agent 编写方案，编码 Agent 消费该方案并实现代码，审查 Agent 依据方案和代码进行审查。
*   **引用源**：[得物技术 - 《从Claude Code实践到团队协作的优化思考｜得物技术》](https://juejin.cn/post/7599828354514567208)

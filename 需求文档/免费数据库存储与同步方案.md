# 免费数据库存储与云端同步方案

面向个人和分享朋友使用的前端学习系统，为解决多端数据同步、防数据丢失以及多用户数据隔离问题，特引入 Supabase 免费云端数据库作为存储服务，并与本地 IndexedDB 组成“离线优先、手动云同步”的混合架构。

---

## 一、 需求分析与目标

1. **多用户数据隔离**：
   - 系统支持多用户使用（主要为个人和分享给朋友）。
   - 每个用户需要有独立的账户体系（采用邮箱免密神奇链接 Magic Link 登录，安全高效，免记密码）。
   - 每个用户只能读写自己的数据（学习进度、错题本、刷题记录等），不能跨用户越权访问。

2. **离线优先与高可用性**：
   - 默认主存储依旧为浏览器本地的 IndexedDB，保证在无网、断网或云端不可用时，系统依然能够流畅使用。
   - 所有刷题记录、宠物状态等高频更新依然直接写入本地。

3. **手动云端同步**：
   - 提供一个用户可见的“云同步”控制面板。
   - 用户可以手动点击“立即同步”按钮，触发本地 IndexedDB 与云端 Supabase 数据库的双向合并。
   - 支持显示上一次成功同步的时间。

4. **安全与防刷**：
   - 数据库接口需要通过 RLS (Row Level Security，行级安全策略) 保证即使 API 密钥暴露在前端，也无法被恶意篡改其他用户的数据。

---

## 二、 技术选型：Supabase

- **数据库**：托管的 PostgreSQL (免费额度 500MB 存储，足够个人和朋友轻量使用多年)。
- **认证**：内置 Supabase Auth (月活 5 万免费用户额度)，采用邮箱免密神奇链接（Magic Link）进行免密鉴权。
- **客户端**：纯前端使用 `@supabase/supabase-js` 直连云端，无需开发与维护后端 API 服务。

## 四、 数据库表结构设计与 RLS 策略 (SQL)

在 Supabase -> **SQL Editor** 中执行以下完整的 SQL 语句，可一键完成建表、启用安全策略（RLS）和为经过认证的用户绑定读写权限：

```sql
-- 1. 用户学习文章进度表 (user_progress)
CREATE TABLE IF NOT EXISTS public.user_progress (
  id text NOT NULL, -- 格式如 'module_id__doc_idx'
  user_id uuid NOT NULL DEFAULT auth.uid(),
  done boolean DEFAULT true,
  updated_at timestamp with time zone DEFAULT timezone('utc'::text, now()),
  PRIMARY KEY (user_id, id)
);

-- 开启行级安全策略 (RLS)
ALTER TABLE public.user_progress ENABLE ROW LEVEL SECURITY;

-- 创建 RLS 策略
CREATE POLICY "Allow individual read/write access to user_progress" ON public.user_progress
  FOR ALL
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);


-- 2. 测验历史记录表 (quiz_records)
CREATE TABLE IF NOT EXISTS public.quiz_records (
  id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id uuid NOT NULL DEFAULT auth.uid(),
  module_id text NOT NULL,
  doc_idx integer,
  score integer NOT NULL,
  total integer NOT NULL,
  pct integer NOT NULL,
  time bigint NOT NULL

ALTER TABLE public.quiz_records ENABLE ROW LEVEL SECURITY;
```

## 五、 手动双向同步算法设计

同步服务定义在 [syncService.js](file:///Users/neoyuan/Desktop/aoyi/个人中高级前端题库/学习系统/src/storage/syncService.js) 中，其核心机制如下：

1. **本地数据结构升级**：
   - 修改 `learnDb.js` 中的 `dbPut` 函数，往本地 IndexedDB 存储数据时，如果传入的是个普通对象，则自动注入或更新本地毫秒级时间戳 `updatedAt = Date.now()`。
2. **消灭错题时的软删除**：
   - 错题答对移出时，业务上从物理删除（`dbDelete`）改为带有被删除标记的 `dbPut(db, 'wrongBook', { qid, isDeleted: true, updatedAt: Date.now() })`。
   - `useLearningDb` 在初始化错题本缓存时，过滤掉带有 `isDeleted: true` 标记的数据。
3. **双向比对合并**：
   - **单向只增历史（如答题历史 quiz_records）**：因只做增量插入，以毫秒时间戳 `time` 作为查重标准，两端互补缺失的记录。
   - **可修改型业务表（进度、错题、刷题统计、宠物状态与事件）**：
     - 如果仅本地存在该主键，则上传至云端；仅云端存在，则拉取并写入本地。
     - 若两端均存在：对比本地 `updatedAt`（数值）与云端 `updated_at`（时区时间转换毫秒数）。若 `本地时间戳 > 云端时间戳`，则上传覆盖；反之，则拉取覆盖。
     - 同步完成后，调用 `reloadLocalCaches` 重载 React 全局 State，使 UI 界面无感更新。

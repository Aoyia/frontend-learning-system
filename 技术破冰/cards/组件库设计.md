# 组件库设计

## 1. 它属于哪个知识板块？

```txt
前端工程化
→ 库与组件库设计
→ 公开 API / 产物形态 / 类型导出 / 使用体验
```

==它**不是“能用 npm publish 的就叫组件库”**。组件库设计真正回答的是：**别人怎么按需引入、怎么在不同环境正确解析、怎么获得正确类型、怎么不被你的更新搞炸**。==

## 2. 陌生技术出现后的学习顺序

### 2.1 它解决什么问题？

==**组件库 / 库设计**解决的是“**业务里能跑的代码**”和“**给别人用的代码**”之间的差距。==

业务代码只需要一个团队、一个项目、一个构建工具能跑就行；公开发布的库要面对：webpack / Vite / Rollup / Rsbuild / Bun 各种构建工具、CJS / ESM / 浏览器 / Node / SSR / RSC 各种运行时、不同 React / Vue 版本、不同 TS 配置、按需引入与 Tree shaking、Tree shaking 之后还要保留必要的副作用（CSS）。

==所以库设计的核心**不是写功能**，而是设计**公开 API、产物形态、入口与导出、类型契约和升级路径**，让消费者在**任何主流环境里都能正确解析、按需引入、保留类型、稳定升级**。==

### 2.2 核心流程

```txt
确定目标使用环境（浏览器 / Node / SSR / RSC）
→ 设计公开 API（命名、参数、组合方式、可扩展性）
→ 选择产物形态（ESM / CJS / dts / CSS / assets）
→ 通过 exports 字段声明入口与条件
→ 标注 sideEffects 与 peerDependencies
→ 生成、校验和发布产物
→ 用 Changelog 与版本号传达变更影响
```

### 2.3 关键词清单

1. public API：库对外暴露的函数、组件、类型、CSS 类名等。一旦发布就成了契约。
2. `exports` field：`package.json` 的 exports 字段，声明每个 subpath 在不同条件（`import` / `require` / `types` / `default`）下应该返回哪个文件。
3. dual package：同时发 ESM 和 CJS，避免使用方在不同模块体系下踩坑。
4. dual package hazard：同一个模块被同时按 ESM 和 CJS 加载，导致两份实例与状态错乱。
5. subpath exports：`@org/ui/button` 这样的子路径导入，由 exports 字段精细控制。
6. types condition：在 exports 里同时声明类型入口，避免类型解析丢失。
7. sideEffects：`package.json` 字段，告诉打包器哪些文件有副作用（例如 CSS、polyfill），不能被 tree shake 掉。
8. tree-shakable API：API 设计本身要友好，避免“一个巨型默认导出”阻碍按需引入。
9. peerDependencies：宿主依赖，组件库通常不打包 React / Vue，而是声明对外要求的版本。
10. css extraction：CSS 抽出策略，决定使用方是手动 import 样式、自动注入还是 CSS-in-JS。
11. semver：语义化版本，patch / minor / major 含义要严格遵守。
12. backward compatibility：向后兼容，破坏性变更必须 major 升级并提供迁移指南。
13. deprecation：废弃策略，先 deprecate 留过渡期，再在下一个 major 删除。
14. a11y / i18n / theming：可访问性、国际化、主题，是组件库长期价值的一部分。

### 2.4 一句面试版

==组件库设计的本质是设计**公开 API 与产物形态**，让消费方在不同构建工具与运行时下都能**正确解析、按需引入、保留类型、稳定升级**；关键载体是 `package.json` 的 **`exports`、`sideEffects`、`peerDependencies`** 字段，以及配套的**版本治理与迁移策略**。==

### 2.5 最小 demo / 最小案例

#### 最小可发布的 `package.json`

```json
{
  "name": "@org/ui",
  "version": "0.1.0",
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./button": {
      "types": "./dist/button.d.ts",
      "import": "./dist/button.js",
      "require": "./dist/button.cjs"
    },
    "./style.css": "./dist/style.css",
    "./package.json": "./package.json"
  },
  "files": ["dist"],
  "sideEffects": ["**/*.css"],
  "peerDependencies": {
    "react": "^18 || ^19",
    "react-dom": "^18 || ^19"
  }
}
```

要点：

1. `exports` 同时声明 `types` / `import` / `require`，避免类型丢失和 dual package hazard。
2. 提供 subpath，让消费方可以 `import { Button } from '@org/ui/button'` 实现更细粒度按需。
3. `sideEffects` 把 CSS 排除在 tree-shaking 之外，避免被打包器误删。
4. `peerDependencies` 声明对宿主框架的版本要求，不重复打包 React。

#### 推荐的 API 形态

不友好的 API：

```ts
export default {
  Button,
  Modal,
  Form
}
```

这种默认大对象不利于 tree shaking，使用方拉一个就拉走全部。

更友好的 API：

```ts
export { Button } from './button'
export { Modal } from './modal'
export { Form } from './form'
```

每个组件独立具名导出，配合 ESM 静态分析就能按需引入。

### 2.6 常见坑怎么定位？

#### 类型在某些项目里识别不到

```txt
确认 exports 是否声明了 types 条件
→ 是否使用了 moduleResolution: bundler 或 nodenext
→ 是否漏配了 subpath 的 types
→ 类型文件是否真的随产物发布（files 字段）
```

#### CSS 没被引入或被 tree shake 掉

```txt
sideEffects 是否包含 *.css
→ 是消费方手动 import 还是组件库自己 import
→ SSR 场景下 CSS 是否被注入到 HTML
```

#### 同一个库出现两份实例

```txt
是否存在 ESM 与 CJS 同时被加载
→ peerDependencies 是否被消费方装成两份
→ Monorepo workspace 是否把同一个包链接到了多个不同版本
```

#### 升级一个 minor 版本就把使用方搞炸

```txt
是不是应该升 major 而不是 minor
→ 公开 API 是否在测试中被回归
→ 是否在 changelog 中标注了破坏性变更
→ 是否给出迁移路径或 codemod
```

### 2.7 库的“向后兼容观”

==一旦发布到公开 registry，**所有公开 API 都是契约**，包括函数签名、组件 props、CSS 类名、ARIA 属性、事件名、抛出的错误类型。==

正确的迭代姿势：

```txt
新增能力
→ minor 版本，不影响已有用法

修复 bug 但不影响 API
→ patch 版本

删除字段 / 改签名 / 改默认值
→ 必须 major 版本
→ 提前一个 minor 版本 deprecate
→ 在 changelog / 文档 / 类型注释里写迁移
→ 必要时提供 codemod
```

### 2.8 是否值得深入？

值得深入，尤其是做基建团队、组件库、SDK、低代码或开源时。优先顺序：

1. 先搞清楚目标运行时（浏览器、Node、SSR、RSC）和构建工具的 resolve 行为。
2. 再掌握 `exports` 字段、subpath、types 条件、`sideEffects`。
3. 然后处理 dual package、peerDependencies 与共享依赖。
4. 接着设计 tree-shakable 的 API 形态与 CSS 策略。
5. 最后建立版本治理：semver、changelog、deprecate、codemod、迁移指南。

优先看官方资料：Node `package.json` Specifier Resolution、TypeScript Module Resolution、esbuild / Rollup 的产物配置、Changesets。

## 3. 选择题自测

### Q1

`package.json` 的 `exports` 字段最核心的作用是什么？

A. 声明项目作者
B. 精细控制每个 subpath 在不同条件下解析到哪个文件，避免类型丢失与解析歧义
C. 替代 README
D. 自动跑测试

答案：B

解析：现代库依赖 `exports` 来正确表达入口、子路径与类型条件，省略它会出现各种解析问题。

### Q2

为什么组件库通常把 React 放在 `peerDependencies` 而不是 `dependencies`？

A. 为了让组件库自己更轻
B. 为了让宿主项目使用单一版本的 React，避免运行时出现两份导致 hooks / context 错乱
C. 因为 React 太大不能装
D. 没有差别

答案：B

解析：UI 框架只能在运行时存在一份实例，否则 hook 状态、context、调度都会失效。

### Q3

`sideEffects` 字段为什么常包含 `*.css`？

A. 防止打包器在 tree shaking 时把样式文件当成无副作用代码删除
B. 让 CSS 文件更小
C. 让浏览器更快
D. 用来发邮件

答案：A

解析：CSS 的副作用是“被引入就生效”，tree shake 时必须保留。

### Q4

下面哪种 API 形态对 Tree shaking 最友好？

A. 一个默认导出的大对象，包含所有组件
B. 每个组件用独立的具名导出
C. 全部挂在 `window` 上
D. 用 `eval` 动态执行

答案：B

解析：具名导出配合 ESM 静态分析能精确按需引入。

### Q5

组件库删掉一个公开 prop，正确的版本号变更是？

A. patch
B. minor
C. major，并提前在前一个 minor 版本 deprecate，提供迁移说明
D. 不需要升版本

答案：C

解析：删除公开 API 是破坏性变更，必须 major，并提前给消费方过渡时间。
